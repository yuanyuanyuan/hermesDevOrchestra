三层风险拦截决策矩阵

L1 — Worker 自行决定

代码实现细节、变量命名、格式调整

决策者: Worker 自身
响应: 直接执行
耗时: 秒级
L2 — Reviewer 决策

技术方案选型、API 设计、架构权衡

决策者: Reviewer (claude -p)
响应: 一次性调用
耗时: 秒~分钟
L3 — 用户决策

安全凭证、破坏性操作、产品方向

决策者: 用户 (Jacky)
响应: Gateway 推送 + 等待
耗时: 分钟~小时
永不自动通过

声明式风险策略 (risk.yaml)

policies:
  - pattern: "rm -rf /*"
    level: L3          # 永不自动通过
    approver: user
    timeout: 0

  - pattern: "git push --force"
    level: L3
    approver: user
    timeout: 300        # 5 分钟无响应则 escalate

  - pattern: "修改 CI/CD 配置"
    level: L2
    approver: reviewer
    timeout: 600

  - pattern: "变量重命名"
    level: L1
    approver: self

Dispatcher 循环 (每 60 秒)

1 回收超时 claim（worker 卡死）
2 回收崩溃 worker（PID 不存在）→ 任务回退 ready + rollback++
3 将 parents 全 done 的 todo 任务提升为 ready
4 原子性 claim ready 任务
5 计算背压比率 → 决定是否降速/暂停
6 按 assignee (profile) spawn worker 进程

路径 A: Implementer 主动 block（主流程）

Implementer 遇到 "RS256 vs HS256?"
→
kanban_block("reviewer-needed: ...")
→
Dispatcher 创建 Reviewer 子任务
→
Reviewer 一次性决策
→
unblock + 恢复
优点: 零额外基础设施，消息永不丢失（SQLite 持久化），天然支持重试
触发条件: Implementer 遵守 SOUL.md 指令，主动 block

路径 B: Claude Code AskUserQuestion → defer（兜底）

① Claude Code 调用 AskUserQuestion 工具
② PreToolUse hook 拦截 → 返回 permissionDecision: "defer"
③ Worker 进程退出，stop_reason: "tool_deferred"
deferred_tool_use 包含完整问题内容（含选项）
④ Dispatcher 读取 deferred_tool_use → 路由给 Reviewer 或用户
⑤ 拿到答案后 → claude -p --resume <session-id> 恢复 Worker
⑥ PreToolUse hook 再次触发 → 返回 allow + answers
⑦ Worker 继续执行，完全无感知
优点: 原生 Claude Code 机制，不依赖 prompt 遵守
限制: defer 仅在 claude -p 非交互模式生效，单次 turn 只能有一个工具调用
版本要求: Claude Code >= 2.1.89

全链路可观测性

Observability Plugin（零侵入采集）

def register(ctx):
    # 1. 采集每个 tool call
    def on_post_tool_call(tool_name, params, result):
        trace_db.record(
            task_id=os.environ.get("HERMES_KANBAN_TASK"),
            tool_name=tool_name,
            result_status="ok" | "error",
            duration_ms=result.get("_duration_ms", 0),
        )
    ctx.register_hook("post_tool_call", on_post_tool_call)

    # 2. 会话结束时汇总
    def on_session_end(session_info):
        trace_db.summarize_session(task_id, session_info)
    ctx.register_hook("on_session_end", on_session_end)

采集数据源

Tool Call Trace → post_tool_call hook
Session Summary → on_session_end hook

SRE-Observer 分析流程（人工升级触发）

① kanban_show() → 读取故障任务基本信息
② 查询 trace.db → 最近 3 次 run 的 tool call 序列
③ 读取 worker logs → 终端错误输出与堆栈
④ 读取 task_events → 状态变迁时间线
⑤ 读取 audit logs → 策略拦截记录
⑥ 对比 parent handoff → 上游交付物是否携带缺陷
⑦ 综合生成 根因报告

根因报告 Metadata Schema

{
  "root_cause_category": "environment|code|deployment|external|policy",
  "confidence": "high|medium|low",
  "symptom": "deploy.sh exited 1: DATABASE_URL not set",
  "root_cause": "生产环境缺少环境变量",
  "responsible_profile": "devops-engineer",
  "upstream_fault": null,
  "recommended_action": "补全 DATABASE_URL 后重新部署",
  "trace_anchor": "tool_call_#3 terminal(deploy.sh) exited 1"
}

SRE 触发条件

任务崩溃
outcome = crashed / timed_out / gave_up
反复回滚
rollback_count >= 2
QA 阻塞
block reason 含 regression / critical_bug / security_flaw

多项目并行管理 (10+ 项目)

Board 隔离架构

Hermes Gateway (dispatcher 嵌入)
    │
    ├── Board: project-alpha
    │   ├── T1 running (implementer)
    │   ├── T2 ready (implementer)
    │   └── T3 todo (reviewer)
    │
    ├── Board: project-beta
    │   ├── T1 running (implementer)
    │   └── T2 todo (orchestrator)
    │
    ├── Board: project-gamma
    │   └── ...
    │
    └── Board: project-juliet
        └── ...

每个 board 拥有:
  · 独立的 SQLite 数据库
  · 独立的 workspace 目录
  · 独立的 dispatcher 循环
  · 任务不可跨 board 引用

管理方式

Dashboard GUI → hermes dashboard → Kanban
CLI → hermes kanban --board project-alpha list
Gateway 消息 → /kanban list (Telegram/Discord)
手机通知 → Gateway 自动推送

跨项目经验共享

Memory 全局 MEMORY.md 中的经验对所有项目生效
Skills agent 创建的 skill 全局可用
Session Search 可搜索所有项目的历史会话
Profile SOUL.md 角色定义全局共享

分层经验归档 (.learnings/)

.learnings/
├── <project-name>/       ← 项目私有经验（默认写入位置）
├── _global/              ← 经 curator 审核的跨项目经验
└── _pending/             ← 待审核的跨项目晋升请求

规则:
  · 经验默认写入项目命名空间
  · 只有 cross-project: true 且 curator 审核通过才进入 _global/
  · agent 查找时项目池优先于全局池
