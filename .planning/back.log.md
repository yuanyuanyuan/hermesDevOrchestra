## 一、核心通讯设计方式

AgentDeck 的通讯设计是分层结构化的，从底层管道到上层通道都做了清晰划分：

### 1. 底层通讯底座

- **基础管道**：依然保留最初设计，使用 `tmux send-keys` 实现跨 tmux session 通讯，这是最稳定可靠的跨 Agent 交互方式
- **协议封装**：通过 MCP（Model Context Protocol）封装核心能力，暴露给不同角色的 Agent 使用
- **管理面隔离**：新增 Master API 走 HTTP 协议，和 worker 使用的 MCP 命名空间完全分离，保证管理能力不被普通 worker 误用

### 2. 上层消息通道设计

为了满足不同场景沟通，新增了多类型通道：
| 通道类型 | 用途 | 特点 |
|-------|-----|-----|
| 群组话题 | 跨 Agent 协作讨论 | 支持生命周期管理，resolve 后停止推送，可 reopen |
| 1v1 DM | master 对 worker 的点对点指令 | 避免干扰群组，适合分派任务/单点催促 |
| 升级通道 | master → 人类（Jacky）的通知 | 关键问题才会触发，保证人类只处理必须决策的事项 |

### 3. 额外通讯特性

- 支持图片附件：消息可携带图片路径，适配多模态 Agent
- SSE 事件流：实现长连接心跳、idle 自动断开、背压保护，适配移动端实时推送
- 消息分级推送：info/warn/blocker 三级，blocker 会触发手机响铃提醒
- **防失控设计**：同一个话题内连续出现 3 条及以上纯 ack（"收到""OK"），master 必须立刻打断，防止两个 Agent 无意义循环对话

## 二、不同角色划分与定位

整个系统是清晰的三层协作模型：

```
Jacky (人类用户) 
    ↓  通过AgentDeck控制台提需求、拍板决策
Master AI (AI代理人/管理中枢) ----也就是hermes
    ↓  通过AgentDeck MCP + Master API调度
Workers (业务Agent/执行单元)
```

每个角色都有严格的边界和职责：
1. **Jacky（人类）**
    - 只负责核心节点介入：提交需求、回答澄清问题、对升级问题拍板、审核系统改进提案
    - 不需要日常调度、进度催促，从"AI调度员"变回"产品经理"

2. **Master（管理者Agent）**
    - **核心原则**：
      1. 不直接编写代码，所有代码改动交给 worker 完成
      2. 承担 90% 日常决策：任务拆分、需求分派、进度跟踪、评审推动、验收结题
      3. 明确升级边界：跨项目破坏性变更、技术选型、外部依赖、生产配置等必须交由人类决策
    - 核心价值：解决人类并行盯多个任务容易走神的问题，AI可以同时处理多个任务不疲劳

3. **Workers（执行Agent）**
    - 一般按业务领域划分，比如项目里的后端、前端、算法、算子等不同 worker
    - 只负责接收任务、完成编码/实现，不做管理决策
    - 通过 MCP 调用基础通信能力，只能看到开放给执行层的工具集

## 三、Master Agent 的 Skill 与生命周期设计

Master 不是单一的大模型聊天，而是**基于 Skill 阶段化设计**，每个阶段有明确的输入和产出：

| 阶段 | 核心产出与动作 |
|-----|-------------|
| **1. 接需求** | 输出 `requirement v0` + 澄清问题清单 |
| **2. 需求澄清** | 输出定稿 `requirement` + 执行 `plan` + 破坏性变更预通知 |
| **3. 任务分派** | 创建 task group + 发送 kickoff 启动消息，给每个worker单独开1v1频道 |
| **4. 进度监督** | 定时检查进度，卡点主动推动，连续无进展自动升级给人类 |
| **5. 成果评审** | 给出结构化结论：`approved`/`changes`/`escalate`，必须附带明确理由 |
| **6. 项目结题** | 生成 `retrospective` 复盘报告 + 追加到历史事实表 + 触发自动改进 |
| **7. 系统优化** | 产出 `improvement proposal` 改进提案，等待人类审核采纳 |

### 关键设计策略

1. **主从分离架构**
   - 主 session：只做轻量路由（接通知、查状态、发消息），保证上下文始终轻量，不会注意力漂移
   - subagent：处理重推理任务（需求起草、计划制定、评审、复盘），借用 Claude 1M 上下文做隔离

2. **文件优先原则**
   - 以文件作为唯一事实源，LLM 上下文只作为工作内存
   - 重启/清理上下文后，会重新读取所有活跃任务的需求和计划，保证信息不丢、不混乱
   - 人类临时的吐槽、半成品想法不会被同步给 worker，只有定稿文件才会被下发，保证执行侧信息干净

3. **强制评审关卡（Review Gate）**
   - 所有关键产出（需求、计划、复盘、改进）都需要独立 `reviewer subagent` 把关
   - 结论必须分三档，且附带明确理由，禁止无理由通过（比如单纯"OK""没问题"会被直接打回）

4. **半自动进化机制**
   借鉴了 `locked-eval + mutable-artifact` 分层思想：
   - **锁定层（不可自动修改）**：评估框架、评审检查清单、失败模式字典
   - **可变层（走流程后可修改）**：通讯协议、worker 画像、任务拆分模板
   - **事实表（仅追加不修改）**：所有项目的历史记录都追加保存，用于归纳改进
   - 每次结题自动触发改进，生成提案后必须等待人类 `adopt` 才会落地，防止AI自我强化错误判断

## 四、工具与能力设计总结

### 对外暴露的工具集

1. **给 Worker 的 MCP 工具（共10个，最初是6个）**：
   基础通信类6个 + 调度类4个（新建Agent、等待上线、列出在线Agent、终止worker），同时有安全白名单限制：spawn 命令只允许指定命令、目录、最大并发，kill 只能作用于worker前缀的Agent，避免误操作。

2. **给 Master 的管理 API（共8个endpoint）**：
   获取 tmux 历史输出、读取Agent细粒度状态、规范化升级通知、组管理、Agent画像管理、send-keys 干预操作等，走HTTP独立部署，和worker隔离。

-------


根据文档梳理，AgentDeck + Master Agent 完整的端到端工作流可以分为**需求启动、过程管控、交付闭环、系统优化**四个大阶段，每个阶段的动作和产出都非常清晰：

---
### 一、需求启动阶段

1. **需求提交**：人类用户通过AgentDeck控制台（电脑网页或手机PWA）提交需求，比如"给监控模块加一个过滤能力，前端展示，后端暴露接口"
2. **需求初处理**：Master AI 起草初始版本需求文档 `requirement v0`，并列出需要人类确认的澄清问题
3. **人类澄清**：用户回答澄清问题，补充边界信息
4. **计划输出**：Master 生成定稿的需求文档和执行方案（明确后端/前端/算法各模块的分工、接口定义），同时识别是否存在跨项目破坏性变更，提前通知人类

### 二、任务分派阶段

1. **创建协作空间**：Master 创建对应的 task group 话题
2. **拉取执行成员**：根据计划把对应业务领域的 Worker Agent 拉入分组
3. **一对一沟通**：为每个 Worker 单独创建 1v1 DM 频道，用于点对点指令同步，避免干扰群组
4. **启动任务**：发送 kickoff 启动消息，明确每个 Worker 的交付要求和截止期望
5. **定时任务注册**：Master 为当前任务注册定时检查，间隔为10-30分钟不等，用于后续进度监控

---
### 三、过程管控阶段

1. **自动进度检查**：Master 按预定时间自动唤醒，读取每个 Worker 的状态
   - 正常推进：不干预，继续等待
   - 空闲超时但未交付：在对应 1v1 频道主动询问卡点问题
   - 连续无进展：三次检查都没有实质进展，自动升级通知人类介入
2. **状态识别能力**：通过解析 tmux 输出，能自动识别以下状态，并做对应处理：
   - 活跃/空闲：正常轮询检查
   - 等待审批/等待输入：主动提醒人类/前置模块
   - 被限流：解析限流解除时间戳，自动预设定时发送"继续"指令，不需要人类蹲点

3. **协作干预**：
   - 如果群组里出现连续3条及以上无内容的纯确认消息（"收到""OK""继续"），Master 会立刻打断，提醒禁止无意义循环对话
   - Worker 之间自行对齐接口等细节，Master 不干预，只做监控

### 四、评审交付阶段

1. **交付触发**：Worker 提交成果后，自动触发结构化评审
2. **独立评审**：由独立的 `reviewer subagent` 做评审，结论分三档，必须附带明确理由：
   - `approved`：通过，进入下一环节
   - `changes`：不通过，列出具体问题和修改方向，打回给 Worker 修改
   - `escalate`：无法决策，升级给人类拍板
3. **循环迭代**：打回修改后重复评审，直到通过或升级

### 五、项目结题阶段

1. **生成复盘**：所有任务环节通过后，Master 自动生成 `retrospective` 复盘报告，总结本次项目中的卡点、经验
2. **记录事实**：把本次项目的关键信息（需求、计划、结论、改进点）追加写入全局事实表，用于后续优化
3. **触发自动改进**：触发 `auto-improve` 流程，subagent 分析复盘和历史事实，归纳可以优化的点
   - 如果发现可改进点，生成 `improvement proposal` 改进提案，推送给人类审核
   - 无改进点则直接结项
4. **推送通知**：手机推送结项通知，告知人类项目已完成，以及是否有待审核的改进提案

### 六、系统优化阶段

1. **人类审核**：用户查看改进提案，判断是否采纳
2. **落地生效**：人类 `adopt`（采纳）后，打标签正式更新到可变层配置（如 Worker 画像、拆分模板），下次任务自动生效
3. **不采纳则丢弃**：如果不采纳，提案保留在目录中不生效，不影响现有流程

---
一个典型完整项目下来，人类用户的主动操作通常只有三次：**提需求 → 回答澄清问题 → 审核采纳改进提案**，从任务分派到进度催促再到评审复盘，全部由 Master AI 自动完成。
