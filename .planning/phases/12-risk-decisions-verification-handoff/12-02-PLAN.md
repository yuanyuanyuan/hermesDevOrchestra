# Phase 12 Plan 02: Local Decision Fallback & Blocking Integration

## Objective

- **What:** Implement adapter-owned local decision commands and wire the watcher/status flow so L3/L4 escalations wait for explicit user approval or rejection.
- **Why:** SAFE-02, DEC-01, and DEC-02 require a no-remote fallback that cannot auto-approve through Hermes, Claude, Codex, timeout, or fallback behavior.
- **Output:** `orch-decisions`, `orch-approve`, `orch-reject`, pending decision state, bus-loop blocking/unblocking, status visibility, and installer wiring.

## Threat Model

<threat_model>

### Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Escalation envelope -> Pending decision | Agent-authored risk context becomes user-actionable approval state. |
| User CLI -> Runtime decision envelope | Human approval/rejection enters the file bus and can resume or stop Codex. |
| State pending queue -> Audit | One-time decision metadata becomes durable evidence. |

### STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-12-05 | Spoofing | `orch-approve` / `orch-reject` | mitigate | Resolve approval IDs only from State pending queue; write `author: "user"` and require a matching active pending request. |
| T-12-06 | Tampering | Pending decision files | mitigate | Validate `used_at`, `expires_at_epoch`, `project_id`, and `task_id` before writing `claude-decision.md`. |
| T-12-07 | Repudiation | User decisions | mitigate | Append `decision_requested`, `decision_approved`, and `decision_rejected` audit records with `approval_id`, `task_id`, and `escalation_id`. |
| T-12-08 | Elevation of Privilege | Claude-generated L3/L4 decisions | mitigate | `orch-bus-loop` rejects L3/L4 continuation unless the decision envelope is user-authored and bound to an approved active `approval_id`. |
| T-12-09 | Denial of Service | Expired pending approvals | mitigate | Expired approvals fail closed and require a fresh pending request; timeout never becomes approval. |

</threat_model>
