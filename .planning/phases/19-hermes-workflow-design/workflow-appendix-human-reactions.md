## 附录 E：人的真实反应

> **架构说明（2026-05-11 更新）：** 本文档中的系统行为由"外部 CLI 引擎"模式执行。block/notification 机制不变，但 findings 来自 CLI 引擎输出。详见 [EXTERNAL-CLI-ENGINE.md](./EXTERNAL-CLI-ENGINE.md)。

### E.1 延迟响应 — 在开会时收到 block 通知

**【场景】**

10:44，Implementer block T2（token 旋转方案选择）。Gateway 向 Jacky 推送 Telegram 消息：

```
🚧 Project Alpha — 任务阻塞，需要决策

任务: t_alpha_002 (实现 HTTP 接口)
问题: token 旋转方案选择
选项: A) 简单滑动窗口  B) 完整旋转  C) 混合方案
```

**【Jacky 的真实状态】**

Jacky 正在和一个潜在客户的 Zoom 会议中。手机在口袋里震动了一下，但他没看。

**【2 小时后】**

12:44，会议结束。Jacky 打开手机，看到 3 条未读消息：
1. 10:44 — T2 blocked（token 旋转）
2. 11:15 — T2 still blocked（系统自动提醒）
3. 11:45 — T2 still blocked（系统第二次提醒）

**【Jacky 的心理活动】**

> "又是 block... 今天第 4 个了。
> 让我看看... token 旋转？这个我记得看过。
> A 是最简单的，B 最安全，C 是混合...
> 
> 我现在很累，不想想太多。选 A 吧，简单滑动窗口。
> 反正 MVP 阶段，够用就行。"

**【后果】**

- Jacky 选择了最简单的方案 A
- 3 个月后产品增长，滑动窗口方案无法支撑并发刷新，需要重写
- 技术债务积累

**【系统应如何支持】**

- Gateway 通知应支持"延迟提醒"：如果用户 30 分钟未响应，在 1 小时后再次提醒，并附带更多上下文
- block 消息应支持"默认选项"：如果用户超时未响应，可按预设默认选项自动继续（仅对非 L3 项目）
- 每日 block 汇总：而不是每条单独推送，可在用户方便的时间（如午休）发送汇总

### E.2 信息过载 — 面对 8 个审查 findings 直接跳过

**【场景】**

Tech-Reviewer 完成 T4 审查，输出了 8 个 findings：
1. `TokenClaims` 缺少 `aud` 声明（低）
2. `generate_token` 的密钥长度检查不完整（中）
3. `verify_token` 的 `exp` 验证未显式开启（**高**）
4. 测试用例缺少 token 篡改测试（中）
5. 文档字符串有拼写错误（低）
6. `refresh_access_token` 未检查 jti 黑名单（中）
7. 错误类型命名不一致（低）
8. `Cargo.toml` 依赖版本未锁定（低）

**【Jacky 的真实反应】**

Jacky 收到审查报告推送：

> "8 个问题... 让我看看。
> 
> （快速浏览）
> 1. aud 声明？好像不是必须的...
> 2. 密钥长度？应该够了吧...
> 3. exp 验证... 这个看起来重要？但 reviewer 标记的是'高'...
> 4. 测试... 后续补吧...
> 5. 拼写错误... 不重要
> 6. jti 黑名单... 什么黑名单？
> 7. 命名不一致... 小问题
> 8. 版本锁定... 以后再说
> 
> 这些都看起来不是阻塞性的。让我直接批准，创建修复任务让 Implementer 处理。"

**【Jacky 的实际操作】**

Jacky 没有逐个阅读每个 finding 的详细说明，直接回复：

```
approve T4. create fix tasks for all findings.
```

**【后果】**

- **finding #3（exp 验证未开启）** 是一个真正的安全漏洞，但 Jacky 没有注意到
- 该漏洞流入 T5 修复列表，和其他 7 个低优先级问题混在一起
- Implementer 按顺序修复，可能优先处理"容易修"的低优先级问题，高优先级安全问题被延迟

**【系统应如何支持】**

- Tech-Reviewer 的 findings 应按严重度分组推送："🔴 1 个安全问题需要您确认" + "🟡 4 个建议" + "🟢 3 个 nit"
- 高严重度 finding 应强制要求用户显式确认（不能批量 approve）
- 安全问题应在通知中附带"如果不修复，可能导致的后果"，提高用户的关注度

### E.3 情绪/疲劳 — 第 10 个 block 通知时感到烦躁

**【场景】**

周五下午 17:30，Jacky 已经工作了 10 小时。手机震动：

```
🚧 Project Epsilon — 任务阻塞，需要决策

任务: t_epsilon_003
问题: 第三方 API 限流，需要您的 API key
```

这是今天第 10 个 block 通知。

**【Jacky 的心理活动】**

> "又是 block！今天已经第 10 个了！
> 
> 我不可能每个小决策都自己做。那我雇 AI 是干什么的？
> 
> （烦躁地打字）
> '以后这种小事自己决定，不要问我！'
> 
> 发送。"

**【后果】**

- Jacky 在烦躁状态下给了一个模糊的指令
- Implementer 的 SOUL.md 说"架构决策必须 block"，但 Jacky 说"不要问我"
- 冲突的指令导致 Implementer 陷入两难：遵守 SOUL.md 还是遵守用户的最新指令？
- 可能需要 PM 介入仲裁

**【系统应如何支持】**

- Gateway 应支持"通知批量化"：在非紧急时段（如工作时间外）批量发送 block 汇总，减少通知频率
- 支持"信任模式"：Jacky 可对某些项目/某些类型的 block 预设"自动批准"规则（如"API key 问题自动批准使用我的默认 key"）
- 检测到用户情绪信号（如大写、感叹号、负面词汇）时，Gateway 应降级为"记录但不推送"，避免进一步刺激

### E.4 盲区 — 技术细节看不懂，凭直觉选 A

**【场景】**

Jacky 收到一个 block，关于数据库索引选择：

```
🚧 Project Zeta — 任务阻塞，需要决策

问题: users 表索引策略
选项:
A) 单列索引 on email
B) 复合索引 on (email, created_at)
C) 部分索引 on email WHERE active = true
```

**【Jacky 的心理活动】**

> "数据库索引... 我学过，但不太记得了。
> A 是最简单的，email 是唯一键，单列索引应该够了吧？
> B 和 C 看起来更复杂，但我不确定好处是什么。
> 
> 让我看看系统给的推荐...
> （系统没有明确推荐，只是列出了选项）
> 
> 好吧，选 A。简单、直观、不容易出错。
> 如果将来性能不够，再加索引。"

**【后果】**

- 6 个月后 users 表增长到 500 万行，按 email 查询变慢
- 需要在线重建索引（添加 created_at），服务中断 3 分钟
- 如果当初选 B，可以避免这次中断

**【系统应如何支持】**

- 对每个选项，系统应提供"推荐程度"和"典型场景"说明：
  - A) "推荐度：⭐⭐ 适合 <10万行数据"
  - B) "推荐度：⭐⭐⭐⭐ 适合需要按时间排序的查询"
  - C) "推荐度：⭐⭐⭐ 适合大部分用户不活跃的场景"
- 支持"延迟决策"：Jacky 可选择"我需要 2 小时研究这个问题"，系统暂停任务并提醒
- 支持"专家咨询"：对于数据库/安全/架构问题，block 消息可附带"需要外部专家意见？"选项

### E.5 通知疲劳 — 关闭 Gateway 通知，导致重要 block 被忽略

**【场景】**

Jacky 周末去海边度假，设置了"勿扰模式"。Gateway 的 Telegram 通知被静音。

周六 14:00，Project Theta 的 Implementer 遇到一个 L3 风险拦截（`DROP TABLE`），block 并等待用户决策。

周六 14:00-周日 20:00，系统按 R17 规则，每 30 分钟发送一次"still blocked"提醒。但 Jacky 的手机静音，他没有看到任何消息。

周日 20:00，Jacky 回到家，打开手机：

```
📵 48 条未读消息来自 Hermes Bot
```

**【Jacky 的真实反应】**

> "48 条... 我看不完。
> 
> （快速滚动，全部标记为已读）
> 反正都是 AI 的工作日志，应该没什么紧急的。
> 
> 去洗澡。"

**【后果】**

- L3 风险拦截（`DROP TABLE`）被忽略
- 该任务在 board 上保持 blocked 状态超过 30 小时
- 如果其他任务依赖该任务，整个项目停滞
- 更严重的是：如果 Jacky 没有注意到，其他 worker 可能继续执行依赖该任务的任务，导致数据不一致

**【系统应如何支持】**

- **分级通知**：L3 风险拦截应使用"紧急通道"（如电话/SMS），而非普通 Telegram 消息
- **Escalation**：block 超过 1 小时后，系统应升级通知方式（Telegram → SMS → 电话）；block 超过 4 小时后，应通知备用联系人
- ** digest vs alert 分离**：工作日志（heartbeat、任务完成）走 digest 模式（每小时汇总）；block/L3 拦截走 alert 模式（即时推送）
- **Vacation mode**：Jacky 可设置"度假模式"，指定某些类型的决策可委托给 PM 或预设规则
