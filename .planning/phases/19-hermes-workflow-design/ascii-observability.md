### Observability Plugin 零侵入采集

```
   ┌─────────────────────────────────────────────────────────────────┐
   │                    Hermes Agent v0.13.0                         │
   │                    (宿主平台，不修改)                            │
   │                                                                 │
   │  ┌─────────────────────────────────────────────────────────┐   │
   │  │              官方 Hook 机制                              │   │
   │  │                                                         │   │
   │  │   ┌──────────────┐          ┌──────────────┐           │   │
   │  │   │ post_tool_   │          │ on_session_  │           │   │
   │  │   │ call hook    │          │ end hook     │           │   │
   │  │   └──────┬───────┘          └──────┬───────┘           │   │
   │  │          │                         │                   │   │
   │  │   每次工具调用后触发               会话结束时触发        │   │
   │  │          │                         │                   │   │
   │  └──────────┼─────────────────────────┼───────────────────┘   │
   │             │                         │                       │
   │  ┌──────────┴─────────────────────────┴───────────────────┐   │
   │  │           Observability Plugin (我们写的)               │   │
   │  │           ~/.hermes/plugins/observability/              │   │
   │  │                                                         │   │
   │  │  def on_post_tool_call(tool, params, result):          │   │
   │  │      trace_db.record(                                  │   │
   │  │          task_id=env["HERMES_KANBAN_TASK"],            │   │
   │  │          tool_name=tool,                               │   │
   │  │          result_status="ok/error",                     │   │
   │  │          duration_ms=result.duration,                  │   │
   │  │          timestamp=now()                               │   │
   │  │      )                                                 │   │
   │  │                                                         │   │
   │  │  def on_session_end(session_info):                     │   │
   │  │      trace_db.summarize(task_id, session_info)         │   │
   │  │                                                         │   │
   │  └────────────────────────┬────────────────────────────────┘   │
   │                           │                                    │
   │                           ▼                                    │
   │  ┌─────────────────────────────────────────────────────────┐   │
   │  │           observability_trace.db                        │   │
   │  │           (SQLite 旁路数据库)                            │   │
   │  │                                                         │   │
   │  │  ┌────────────┐ ┌────────────┐ ┌────────────┐          │   │
   │  │  │tool_call   │ │session     │ │task_run    │          │   │
   │  │  │_traces     │ │_summaries  │ │_snapshots  │          │   │
   │  │  └────────────┘ └────────────┘ └────────────┘          │   │
   │  └─────────────────────────────────────────────────────────┘   │
   │                                                                 │
   └─────────────────────────────────────────────────────────────────┘

        大白话:
        ┌──────────────────────────────────────────────────────┐
        │  像给汽车装黑匣子，但不用拆发动机                     │
        │                                                      │
        │  Hermes 官方预留了"插槽"（Hook）                      │
        │  我们插一个插件进去                                   │
        │                                                      │
        │  每次 AI 工人调用工具（跑测试、读文件）                │
        │  → 插件自动记录"调了什么、结果如何、花了多久"         │
        │                                                      │
        │  工人下班时                                          │
        │  → 插件汇总"总共调了多少次、成功失败比例"             │
        │                                                      │
        │  全程不碰 Hermes 源码，插拔自如                       │
        └──────────────────────────────────────────────────────┘
```

**大白话：** 像给汽车装黑匣子，但不用拆发动机。Hermes 官方预留了插槽，我们插一个插件，每次工人调用工具时自动记录，全程不碰源码。

### SRE-Observer 自动触发与分析

```
   ┌─────────────────────────────────────────────────────────────────┐
   │                     故障触发条件（任一满足）                      │
   └─────────────────────────────┬───────────────────────────────────┘
                                 │
              ┌──────────────────┼──────────────────┐
              │                  │                  │
              ▼                  ▼                  ▼
   ┌─────────────────┐ ┌─────────────────┐ ┌─────────────────┐
   │  ① 任务崩溃     │ │  ② 反复回滚     │ │  ③ QA 阻塞      │
   │  crashed        │ │  rollback≥2     │ │  critical_bug   │
   │  timed_out      │ │                 │ │  security_flaw  │
   │  gave_up        │ │                 │ │  regression     │
   └────────┬────────┘ └────────┬────────┘ └────────┬────────┘
            │                   │                   │
            │                   │                   │
            ▼                   ▼                   ▼
   ┌─────────────────────────────────────────────────────────────┐
   │                    ④ DevOps 部署失败                          │
   │                       outcome ≠ completed                    │
   └─────────────────────────────────────────────────────────────┘
            │
            │ 任一触发
            ▼
   ┌─────────────────────────────────────────────────────────────────┐
   │              步骤 1: 自动创建 SRE 分析任务                        │
   │                                                                 │
   │  kanban_create(                                                 │
   │    title="根因分析: T5 部署失败",                                │
   │    assignee="sre-observer",        ← 专属角色                   │
   │    body="T5 crashed，请定位根因",                                │
   │    parents=[T5],                   ← 关联故障任务               │
   │    priority="high"                 ← 高优先级插队               │
   │  )                                                              │
   └─────────────────────────────────────────────────────────────────┘
            │
            ▼
   ┌─────────────────────────────────────────────────────────────────┐
   │              步骤 2: SRE-Observer 执行 7 步分析                  │
   │                                                                 │
   │   ① ──→ kanban_show()                                          │
   │          读取故障任务基本信息                                    │
   │            │                                                    │
   │            ▼                                                    │
   │   ② ──→ 查询 trace.db                                          │
   │          最近 3 次 run 的 tool call 序列                         │
   │            │                                                    │
   │            ▼                                                    │
   │   ③ ──→ 读取 worker logs                                       │
   │          终端错误输出与堆栈                                      │
   │            │                                                    │
   │            ▼                                                    │
   │   ④ ──→ 读取 task_events                                       │
   │          状态变迁时间线                                          │
   │            │                                                    │
   │            ▼                                                    │
   │   ⑤ ──→ 读取 audit logs                                        │
   │          策略拦截记录                                            │
   │            │                                                    │
   │            ▼                                                    │
   │   ⑥ ──→ 对比 parent handoff                                    │
   │          上游交付物是否携带缺陷                                  │
   │            │                                                    │
   │            ▼                                                    │
   │   ⑦ ──→ 综合生成根因报告                                        │
   │          8 层定位模型匹配                                        │
   │                                                                 │
   └─────────────────────────────────────────────────────────────────┘
            │
            ▼
   ┌─────────────────────────────────────────────────────────────────┐
   │              步骤 3: 输出结构化报告                               │
   │                                                                 │
   │  ┌─────────────────────────────────────────────────────────┐    │
   │  │  root_cause_category: "environment"  ← 环境层           │    │
   │  │  confidence: "high"                   ← 高置信度         │    │
   │  │  symptom: "deploy.sh exited 1"       ← 症状            │    │
   │  │  root_cause: "缺少 DATABASE_URL"     ← 根因            │    │
   │  │  responsible_profile: "devops"       ← 责任方          │    │
   │  │  upstream_fault: null                ← 非上游问题      │    │
   │  │  recommended_action: "补全环境变量"   ← 修复建议        │    │
   │  │  trace_anchor: "tool_call_#3..."     ← 证据锚点        │    │
   │  └─────────────────────────────────────────────────────────┘    │
   └─────────────────────────────────────────────────────────────────┘
            │
            ▼
   ┌─────────────────────────────────────────────────────────────────┐
   │              步骤 4: 通知 CEO + Dashboard 更新                    │
   │                                                                 │
   │  ┌─────────────────┐          ┌─────────────────┐              │
   │  │ Gateway 推送     │          │ Dashboard        │              │
   │  │                 │          │                 │              │
   │  │ "Alpha 部署失败  │          │ SRE 报告列表     │              │
   │  │  根因: 缺环境变量│          │ +1 条新报告      │              │
   │  │  建议: 补上重试" │          │ 可筛选查看       │              │
   │  └─────────────────┘          └─────────────────┘              │
   └─────────────────────────────────────────────────────────────────┘
```

**大白话：** AI 工人出事故了，系统自动叫一个 AI 侦探来破案——翻遍所有证据（操作记录、错误日志、时间线、环境快照、上游交接单），最后出一份"事故报告"：谁的问题、怎么修、证据在哪。老板收到推送通知就知道全貌。

### 故障定位 8 层模型

```
   ┌─────────────────────────────────────────────────────────────────┐
   │                    故障定位层次模型                               │
   │                                                                 │
   │  ⑧ 策略层 ───────────────────────────────────────────────┐     │
   │     │  "Risk Policy 拦截了操作"                             │     │
   │     │  定位: audit logs + kanban_block reason               │     │
   │     ▼                                                    │     │
   │  ⑦ 外部层 ───────────────────────────────────────────────┤     │
   │     │  "第三方 API 挂了/网络超时"                           │     │
   │     │  定位: web/browser tool 调用失败记录                   │     │
   │     ▼                                                    │     │
   │  ⑥ 部署层 ───────────────────────────────────────────────┤     │
   │     │  "CI/CD 失败/发布脚本错误"                            │     │
   │     │  定位: DevOps logs + deployment trace                 │     │
   │     ▼                                                    │     │
   │  ⑤ 资源层 ───────────────────────────────────────────────┤     │
   │     │  "磁盘满/内存不足/API 限流"                           │     │
   │     │  定位: df -h / hermes status / 工具耗时异常            │     │
   │     ▼                                                    │     │
   │  ④ 环境层 ───────────────────────────────────────────────┤     │
   │     │  "依赖缺失/配置错误/权限不足"                         │     │
   │     │  定位: Environment Snapshot + git status 对比         │     │
   │     ▼                                                    │     │
   │  ③ 验收层 ───────────────────────────────────────────────┤     │
   │     │  "QA 功能测试未通过/回归 Bug"                         │     │
   │     │  定位: qa-tester block reason + test output           │     │
   │     ▼                                                    │     │
   │  ② 审查层 ───────────────────────────────────────────────┤     │
   │     │  "Reviewer 拒绝通过/安全审计不通过"                   │     │
   │     │  定位: tech-reviewer kanban_complete findings         │     │
   │     ▼                                                    │     │
   │  ① 代码层 ───────────────────────────────────────────────┘     │
   │     "语法错误/测试失败/逻辑缺陷"                                  │
   │     定位: worker logs + tool call trace                          │
   │                                                                  │
   └─────────────────────────────────────────────────────────────────┘

        SRE-Observer 分析时按顺序排查:
        ┌─────────┐
        │ 读取证据 │
        └────┬────┘
             │
             ▼
        ┌─────────┐     是      ┌─────────┐
        │代码层?  │────────────→│ 定位完成 │
        └────┬────┘             └─────────┘
             │ 否
             ▼
        ┌─────────┐     是      ┌─────────┐
        │审查层?  │────────────→│ 定位完成 │
        └────┬────┘             └─────────┘
             │ 否
             ▼
        ┌─────────┐     是      ┌─────────┐
        │验收层?  │────────────→│ 定位完成 │
        └────┬────┘             └─────────┘
             │ 否
             ▼
             ... (继续向上排查)
             │
             ▼
        ┌─────────┐     是      ┌─────────┐
        │策略层?  │────────────→│ 定位完成 │
        └────┬────┘             └─────────┘
             │ 否
             ▼
        ┌─────────┐
        │未知/需人工│
        └─────────┘
```

**大白话：** 侦探破案有 8 个方向——从"代码写错了"到"被安全策略拦住了"，一层一层往上查。先查最可能的（代码问题），再查环境、资源、部署，直到找到真凶。
