# Phase 9 Plan 01: Upstream Hermes Agent Baseline

## Objective
- **What:** Establish the real upstream Hermes Agent baseline, pin it to a commit, document capability gaps, and remove the old standalone Node CLI.
- **Why:** v1.1 must build on `https://github.com/NousResearch/hermes-agent`; local `hermes` runtime code would shadow upstream and recreate the wrong direction.
- **Output:** A capability baseline summary plus repository cleanup that leaves upstream `hermes` as the only `hermes` command.

## Threat Model

<threat_model>
Primary risks:
- **PATH shadowing:** A local `bin/hermes.js` or npm bin could override upstream `hermes`, causing future phases to test the wrong CLI. Mitigation: delete local `hermes` scaffolding and verify no package-level `hermes` bin remains.
- **User-environment overwrite:** Real current-user install could overwrite existing Hermes config. Mitigation: preflight `command -v hermes`, `~/.hermes`, and `~/.hermes-orchestra`; stop on unexpected existing state.
- **Upstream drift:** Tracking `main` without a pin makes later behavior non-reproducible. Mitigation: record exact commit SHA and upgrade procedure.
- **Scope creep:** Filling adapter gaps during Phase 9 would blur discovery with implementation. Mitigation: capability matrix only; implementation deferred to Phases 10-12.
</threat_model>
