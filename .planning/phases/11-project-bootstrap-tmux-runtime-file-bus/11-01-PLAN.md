# Phase 11 Plan 01: Project Bootstrap & Helper Template Foundation

## Objective

- **What:** Extract the installed `orch-*` helper bodies into package template files and make `orch-init` produce a durable, checkable project registration.
- **Why:** Phase 11 needs more runtime behavior than is safe to maintain as large heredocs inside `setup.sh`; project bootstrap must also obey the four-layer State/Audit/Runtime boundary.
- **Output:** Shared shell helper library, template-installed `orch-init`/`orch-status`, updated installer copy behavior, and bootstrap documentation.

## Threat Model

<threat_model>

| ID | Threat | Severity | Mitigation |
|----|--------|----------|------------|
| T-11-01 | Malicious or ambiguous project ID escapes path/session names | high | Enforce `^[a-z0-9][a-z0-9-]{1,30}[a-z0-9]$` before any filesystem or tmux use. |
| T-11-02 | Runtime bus becomes mistaken for durable state | high | Write `project.env`, `paths.json`, and `projects.json` only under State; Runtime holds active bus files only. |
| T-11-03 | Installer overwrites user helper edits without a trace | medium | Keep package templates as source of truth and install idempotently; do not mutate project worktrees except copied `.claude/settings.json`. |

</threat_model>
