> **补全内容**：
> - 目标：为支付网关增加限流保护。

>

**【一阶：方向辩论】**

> **Kimi**：启动方向辩论（标准通道）。
> 
> **dynamic_assembly**：本次涉及 payment_compliance（扩展团队）、security、scalability_arch、api_design、observability、devops_sre。
> 
> **adversarial_debate**（核心争议）：
> - **正方（scalability_arch）**：应加限流。支付网关无熔断保护，存在单点流量冲击风险。建议 Redis + Token Bucket。
> - **反方（business）**：限流可能误杀正常支付请求，黑五/促销期间风险高。建议先加监控告警，人工确认后再限流。
> - **api_design**：429 错误码需前端配合，当前 `payment.py` 的异常处理未覆盖 429，需评估前端影响范围。
> - **payment_compliance**：限流日志不得包含卡号后 4 位以外的任何支付敏感数据。
> - **observability**：用户约束"不要影响现有支付成功率监控"需要明确定义——429 是"业务拒绝"还是"系统错误"？这会影响监控分类。
> 
> **jury_panel 结论**：方向可行，但需满足以下条件才继续：
> 1. 限流阈值支持环境变量动态配置（非硬编码）。
> 2. 429 响应统一走 `HTTPException(429, detail="rate_limited")`，便于前端识别。
> 3. 监控分类方案：429 不计入支付成功率分母，也不计入系统错误；单独新增 `payment_rate_limited` 指标。
> 4. 黑五期间可开关限流（feature flag）。
> 
> 方向报告置信度：高。以上 4 项条件是否接受？如接受，自动进入二阶。

| 维度 | 评分 | 说明 |

## 用户心路历程总曲线

```
阶段        0阶       一阶       二阶       三阶       四阶       五阶       六阶
情绪曲线    🟡→       🟢→       🟢→       🔴→       🟢→       🟡→       🟢
           震惊/     紧张/     满意      焦虑/     惊喜/     轻微     信任
           安心      释然                不安      信任      不安     沉淀
           ↑                         ↑                   ↑
        信息量大                  执行静默              残余风险
        但有用                    制造焦虑              是否被忽略
```

**核心发现**：

1. **信任建立点**：0阶补全质量（尤其是隐性约束）、一阶的"条件清单"、四阶的自动修复。
2. **信任损耗点**：三阶执行无反馈、错误纠正时"被教育感"。
3. **用户最满意的时刻**：不是系统"做得多"，而是系统"在关键处帮我考虑到了我没想到的"（如黑五场景、PCI 合规）。
4. **用户最焦虑的时刻**：不是系统"问得多"，而是"突然没声音了，不知道在干嘛"。

## 可行性总体 Verdict

| 维度 | 结论 |
|------|------|
| **六阶闭环完整性** | ✅ 可行。0阶→六阶逻辑自洽，每阶段有明确输入/输出和决策门。 |
| **通道分级合理性** | ✅ 可行。快速/轻量/标准三层区分清晰，渐进式校准策略降低初期误判风险。 |
| **自动推进 vs 用户确认平衡** | ⚠️ 基本可行，但需调整。快速通道应更少确认；三阶需增加进度可见性；五阶 warnings 需主动摘要通知。 |
| **错误纠正机制** | ✅ 机制有效。两轮 + Override 留痕形成合理软约束。但第一轮话术需更克制，避免对抗感。 |
| **信息保真** | ✅ 设计合理。来源/置信度/验证方式三层标注，Conflict Ledger 避免静默覆盖。实际效果依赖 Gateway 实现严谨度。 |
| **用户心理接受度** | ⚠️ 需要打磨。系统"聪明"会让用户惊喜，但"话多"或"静默"都会损耗信任。建议增加"对话密度"控制（verbose/compact/silent）。 |

## 对 PRD 和实现的具体改进建议

### 1. 新增"执行心跳"机制（高优先级）

三阶执行期间，Gateway 每 30 秒向用户推送一次结构化进度：

```json
{
  "stage": "implementation",
  "elapsed_seconds": 120,
  "tasks": {
    "completed": 2,
    "in_progress": 1,
    "pending": 2
  },
  "current_activity": "Claude 审查中，预计 30 秒",
  "blocked": false
}
```

### 2. 快速通道默认自动合并（高优先级）

在 `project-profile.yaml` 中增加：

```yaml
quick_channel:
  auto_merge: true  # 默认 true，用户可关
  notification: "silent"  # silent/compact/verbose
```

### 3. 错误纠正"渐进展开"模式（中优先级）

第一轮只展示：
> ⚠️ 此操作存在风险：[一句话]。建议：[替代方案]。是否查看详细原因？

用户点击"查看"才展示同类历史、规范原文等。

### 4. 0阶"摘要/详细"双模式（中优先级）

用户可设置默认模式：
- **摘要模式**："推测文件为 `Login.tsx`，i18n key 已存在。是否有误？【是/修正/查看详细】"
- **详细模式**：当前完整补全包。

Sprint 3 CLI 模拟补充：

- `orch-mvp-wizard --project-dir <dir>` 会直接按 `interaction.default_mode` 输出摘要或详细确认视图，不再强制进入完整安装向导。
- `orch-mvp-wizard --interactive --mock` 采用两轮门控：第一轮给出概要选项，选择 `Explain` 后进入第二轮细分追问。
- `orch-mvp-wizard --batch` 会明确打印 `non-interactive: two-round correction degraded to single-round confirmation`，避免假装执行了两轮纠错。

### 5. 五阶残余风险主动通知（中优先级）

`pass_with_warnings` 自动通过后，给用户推送：
> ✅ 任务已完成。
> 
> 📋 残余风险（低）：Redis 超时 0.5s，极端网络抖动时可能误判。
> 📋 建议：黑五期间关注 `payment_rate_limited_total` 指标。
> 
> 完整报告：[链接]

### 6. Sprint 4 Quick → Light → Standard 渐进模拟

| 场景 | 输入 | 预期通道 | 依据 |
|------|------|----------|------|
| Week 1 单文件 i18n key 修正 | `task_type=i18n`, `files_count=1` | Quick | 低风险、单文件、机械证据可验证 |
| Week 1 多文件重构 | `task_type=refactor`, `files_count=5` | Standard | Week 1-2 禁止 Quick 多文件修改 |
| Week 3 单文件重构 | `task_type=single_file_refactor`, `files_count=1` | Quick | Week 3 校准节奏允许单文件重构 |
| Week 4 三文件重构 | `task_type=refactor`, `files_count=3` | Quick | Week 4+ 允许不超过 `channels.quick.max_files` 的重构 |
| Quick kill switch 打开 | `channels.quick.enabled=false` | Light 或 Standard | 原 Quick 候选降级并写入 `logs/channel-routing.jsonl` |
| 校准证据不足 | `confidence=0.5`, `coverage=0.3` | Standard | Rollout Gate 返回 `forced_standard: true` |
