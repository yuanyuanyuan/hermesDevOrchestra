# Worker Capability Negotiation Uses Explicit Fallback

The full worker system will stage full target registries at `config/workers/full/backends.json` and `config/workers/full/roles.json` while the root worker configs remain MVP/current runtime inputs until cutover. Gateway must validate requested worker pairings through registered roles, backend availability, required capabilities, protocol support, workspace/session support, risk ceiling, and fallback policy. If a requested backend is unavailable or incompatible, Gateway must not silently substitute another backend: it writes or returns a `capability_negotiation_report` and blocks for Kimi unless the role registry explicitly lists a safe fallback, the failure class is allowed, and no risk, security, release, parallel, rule-change, or authority boundary forbids fallback.
