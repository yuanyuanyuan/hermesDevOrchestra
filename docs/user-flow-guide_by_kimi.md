# Hermes Dev Orchestra 用户流程指南

## 1. 一句话理解

Hermes Dev Orchestra 的根本目的不是“多代理很复杂地跑一遍流程”，而是在收到一个需求/任务后，自动补齐执行所需的上下文、依赖、隐性约束、验收标准和清晰 prompt，并尽量减少信息丢失、误读和错误传播。

用户日常只需要提供：

1. 目标：想要什么结果。
2. 约束：什么不能做，或必须遵守什么边界。
3. 线索：可用资料、背景、相关文件、已有判断。

系统默认自动完成后续的需求补全、方向论证、方案拆解、执行、测试、修复、全局评估和经验沉淀。只有出现危险操作、任务偏离、冲突无法解决、低置信度关键判断时，才上浮给用户确认。

## 2. 各角色分工匹配度验证

| 角色 | 职责定义 | 知识库设计规范 | 匹配度 |
|------|----------|----------------|--------|
| **用户** | 提供目标、约束、线索，并在例外情况下确认 | 意图驱动范式下，人类不规定“怎么做”，只在危险、偏离、冲突、低置信度时决策 | 完全匹配 |
| **Kimi** | 需求补全 + 编排 + 调度 + 验收 + 审计 | Kimi 是上层 Orchestrator，负责意图转换、上下文补全、任务拆解、双重验收、经验审计 | 完全匹配 |
| **Gateway** | 翻译适配、状态投影、证据门控、冲突记录 | Gateway 是通信和权限边界，负责格式转换、消息路由、依赖/证据投影、风险门控 | 完全匹配 |
| **Hermes agents** | 下层执行、多 Agent 生命周期和依赖推进 | Hermes Execution Framework 管理任务持久化、上下文隔离、工具调用、辩论团队和执行生命周期 | 完全匹配 |
| **Claude/Codex** | 具体编码、审查、测试、专项审计 | 底层执行模型负责工程动作和证据产出，符合“判断在上层，执行在下层”的分工 | 完全匹配 |

关键边界：

- Kimi 是补全者和裁决者，不直接改 Hermes Kanban。
- Gateway 是状态和证据权威边界，不替代用户批准高风险事项。
- Hermes agents 负责执行、隔离、生命周期和证据产物。
- Claude/Codex 只在被调度后做具体编码、评审、测试、审计，不能绕过 Gateway 推进状态。

## 3. 默认自动推进原则

系统默认自动推进，而不是每个阶段都让用户确认。

| 情况 | 处理方式 |
|------|----------|
| 信息足够、风险低、证据完整、无冲突 | 自动进入下一阶段 |
| 验收标准缺失但可从上下文可靠推断 | 自动补全，并标记为系统推断 |
| 存在多个合理方案但风险低 | Kimi 选择推荐方案，并保留备选方案和理由 |
| 快速通道任务完成且证据通过 | 自动合并并通知用户，无需等待确认 |
| 涉及 L3/L4、生产发布、权限、密钥、CI/CD、根规则 | 必须上浮用户审批 |
| 方向或方案与原始目标偏离 | 阻塞并请求 Kimi/用户裁决 |
| 关键依赖、测试结果、审查结论互相冲突 | 进入冲突清单，无法自动消解时上浮 |
| 关键判断低置信度 | 暂停推进，请用户补充或确认 |

这意味着“点确认”不是常规步骤，而是异常处理机制。

**Run 生命周期异常状态的用户交互**：

| 状态 | 用户看到的提示 | 用户可执行的操作 |
|------|---------------|-----------------|
| `blocked` | "⏸️ 任务已阻塞：[具体原因，如'等待 L3 审批'、'检测到 write scope 越界']" | 查看阻塞详情、补充信息、发起审批、选择回滚、取消 Run |
| `paused` | "⏸️ 任务已暂停（由您或系统发起）" | 继续执行、取消 Run、查看当前进度快照 |
| `cancelled` | "❌ 任务已取消" | 查看已产出产物（如有）、重新创建 Run |
| `rollback_requested` | "↩️ 回滚请求已发起，正在撤销变更……" | 查看回滚进度、确认回滚完成、或中断回滚转为接受当前状态 |

## 4. 用户错误纠正机制

用户可能是错误的。系统不能无条件服从，也不能傲慢反驳。需要一套“客观、有证据、留面子”的纠正机制。

### 纠正的边界

系统只纠正**客观错误**，不纠正**用户偏好**。

| 类别 | 示例 | 系统行为 |
|------|------|---------|
| **客观错误** | “用明文存密码”、“跳过测试直接上线”、“在生产环境直接改数据” | 必须纠正，提供证据；用户坚持则上浮审批或记录 Override |
| **实现路径分歧** | “用 Redis 做缓存” vs 系统建议“本地 LRU” | 不纠正，作为二阶辩论输入，让用户看证据后决策 |
| **用户偏好** | “我喜欢单引号”、“函数命名用驼峰” | 服从，写入 SOUL.md 作为项目风格 |
| **信息不足导致的误判** | “这个改动没有风险”（实际有） | 纠正，提供风险证据，不评价用户能力 |

### 纠正的话术原则

**三要素**：客观陈述事实、引用证据或规则、给出替代方案。

**禁止**："你错了"、"这很明显不对"、"作为开发者你应该知道"。

**推荐**："这个方案存在 X 风险，依据是 Y。建议改为 Z，原因是 W。"

### 纠正的渐进展开模式

为避免用户在首轮纠正时感到"被系统教育"，纠正信息按轮次渐进展开：

- **第一轮（极简）**：只给一句话风险概述 + 一个替代方案。详细证据默认折叠，用户可主动展开查看。
- **第二轮（完整）**：若用户坚持，再展示同类历史事故、安全规范原文和完整风险证据链。

**交互 UX 细节**：

第一轮消息格式（聊天/面板通用）：
```text
⚠️ 跳过测试直接部署支付代码存在高风险。
建议：走紧急通道（3 分钟最小测试集）。
[接受建议] [坚持原方案] [查看详细原因 ▼]
```
- 点击【查看详细原因 ▼】后，在当前消息下方**原地展开**折叠区域（不发送新消息），展示：
  - 风险等级标签（L3 / L4）
  - 同类历史事故摘要（1-2 条，含日期和影响面）
  - 相关规范原文引用（如 AGENTS.md 条款）
  - 【收起 ▲】按钮
- 用户选择【接受建议】→ 系统立即替换为替代方案，记录为 `corrected`，不进入 Override。
- 用户选择【坚持原方案】→ 系统记录为 `user_insisted_round_1`，进入第二轮。

第二轮消息格式（仅当用户坚持时触发）：
```text
⚠️ 您仍坚持跳过测试。以下是完整风险证据：
━━━━━━━━━━━━━━━━━━━━
同类历史事故：
• 2025-11 未测试限流导致黑五误拦截 12% 订单（P0 事故）
• 2024-06 支付回调未覆盖空指针，导致重复扣款 37 笔

规范原文：
• AGENTS.md 安全红线 SR-3：生产代码必须通过最小测试集
• SRE 手册 5.2：支付流变更的测试豁免仅由 CTO 审批

建议替代方案：
• 方案 A：走紧急通道（3 分钟最小测试集）
• 方案 B：标记为热修复，先灰度 1% 流量并自动回滚
━━━━━━━━━━━━━━━━━━━━
[接受方案 A] [接受方案 B] [仍坚持原方案]
```
- 若用户选择【仍坚持原方案】：
  - 系统判定风险等级：若为 L3/L4，Gateway 阻塞并推送审批卡片（@审批人）；若为非高风险，记录 Override 并附证据，继续执行。
  - Override 记录以**可折叠摘要**形式追加到当前对话底部，用户可随时展开查看审计 ID 和证据引用。

### 纠正的 Escalation 路径

1. **第一轮纠正**：极简风险提示 + 替代方案，详细证据折叠。
   - 用户接受 → 继续正常流程。
   - 用户坚持 → 进入第二轮。
2. **第二轮纠正**：展示完整证据（同类历史事故 / 安全规范原文）。
   - 用户接受 → 继续。
   - 用户仍然坚持 → 判定为 L3/L4 风险则 Gateway 阻塞要求审批；非高风险则记录为“用户 Override”并附证据，继续执行。
3. ** Override 留痕**：所有 Override 必须进入审计记录，六阶复盘。同一用户同类错误 Override 3 次以上，六阶建议加强该领域培训或调整权限。

## 5. 工作流总览：0阶 + 六阶闭环 + 通道分级

| 阶段 | 核心目标 | 用户动作 | 系统动作 | 输出物 |
|------|----------|----------|----------|--------|
| 0阶 | 需求补全与上下文水合 | 说目标、约束、线索 | 自动补齐环境、上下游、依赖、隐性内容、验收矩阵和 prompt | 《需求补全包》 |
| 一阶 | 方向辩论 | 仅在低置信度/冲突/高风险时确认 | 判断方向是否值得做，暴露业务和风险问题 | 《最佳选择报告》 |
| 二阶 | 方案辩论 | 仅在高风险或不可逆方案时确认 | 锁定技术路线、DAG、并行边界和测试策略 | 《具体实现报告》 |
| 三阶 | 具体执行 | 通常无需介入 | Claude/Codex 编码、测试、审查；Gateway 收证据 | 《任务反馈文档》 |
| 四阶 | 改进实现 | 仅在越权或范围扩大时确认 | 根据反馈当场修复并复测复审 | 《改进报告》 |
| 五阶 | 全局评估 | 仅在残余风险超阈值时确认 | 多维度验收质量、安全、性能、文档、可维护性 | 《整体改进报告》 |
| 六阶 | 持续改进 | protected target 变更时确认 | Kimi 审计日志，Hermes 落地流程/规则优化 | 《工作流优化建议》 |

### 通道分级

不是所有任务都需要完整六阶。系统根据规则引擎初筛 + 简单辩论确认，分三层通道：

| 通道 | 跳过阶段 | 审查深度 | 典型场景 |
|------|---------|---------|---------|
| **快速通道** | 一阶、二阶 | light | 纯文案、单文件 typo、配置值调参 |
| **轻量通道** | 一阶、二阶 | standard | 简单逻辑修改、无风险 UI 调整、文档更新 |
| **标准通道** | 无 | deep | 其他所有任务 |

快速通道判定需经一轮极简辩论确认（1-2 支团队，1 轮，30 秒内），不可直接黑盒执行。

## 6. 0阶：需求补全与上下文水合

0阶是这套系统最重要的入口。它解决的问题是：用户给出的需求通常不完整，直接执行会造成误读、漏依赖、漏测试、漏验收。

对于**新项目首次使用**，0阶还包含项目接入探测：

1. Kimi 自动读取文件树、依赖文件（`package.json`/`pyproject.toml`/`go.mod` 等）、CI/CD 配置、测试目录结构。
2. 生成《项目探测报告》草稿，展示推测出的技术栈、测试命令、部署目标、风险标志。
3. **开发者确认或修正（交互 UX）**：
   - 系统以**结构化卡片**展示探测结果，每项字段附带【✅ 正确 / ✏️ 修正】按钮。
   - 高置信度字段（语言、框架）默认折叠详细来源，显示绿色勾；用户可展开查看推断依据（如"来自 `pyproject.toml` 的 `requires-python = "3.12"`"）。
   - 中低置信度字段（数据库、部署目标、风险标志）以黄色问号标记，必须用户显式确认；点击【✏️ 修正】后弹出下拉框或输入框，提供常见选项（如数据库：PostgreSQL / MySQL / MongoDB / 其他）。
   - 底部显示进度条（如"已确认 5/8 项"）和【一键确认剩余高置信度项】按钮。
   - 若用户在 5 分钟内未响应，系统推送一次提醒；超过 10 分钟未响应，Gateway 将未确认项标记为 `pending_verification`，允许后续在常规任务中再补全，不阻塞首次接入。
4. 生成 `.hermes/project-profile.yaml`、初始 `AGENTS.md` 和 `SOUL.md`。
5. 完成接入，进入正常工作流。

对于**已有项目**，Kimi 直接基于 `project-profile.yaml` 和已有规则自动补齐：

- **环境依赖**：运行环境、工具链、版本、密钥、服务、权限、部署目标。
- **上游依赖**：API、数据源、配置、外部服务、已有模块、前置任务。
- **下游影响**：调用方、UI、测试、部署、文档、监控、用户流程。
- **代码依赖**：模块、接口、schema、迁移、构建链路、共享类型。
- **隐性内容**：未明说但必须满足的安全、合规、性能、兼容、可维护性要求。
- **验收矩阵**：每条需求对应的验收标准、测试方式、证据 artifact 和责任角色。
- **执行 prompt envelope**：给 Hermes/Claude/Codex 的完整任务上下文，包含目标、非目标、约束、依赖、验收和失败策略。

输出《需求补全包》，至少包含：

- 原始用户意图。
- 系统补全内容。
- 已验证事实。
- 未验证假设。
- 依赖图。
- 冲突清单。
- 验收矩阵。
- 执行 prompt envelope。
- 每个顶层区块都必须附 `source_input_hash` 与 `projection_timestamp`，关键结论节点必须附 `source` / `confidence` / `verification_method`。
- 对模糊输入，系统输出低置信度补全包并进入人工确认，而不是用空字段伪装成完整结果。

如果关键字段无法可靠推断，系统才向用户提问。

### 0阶输出模式：摘要 vs 详细

0阶补全包信息量大，但用户日常只关心"推测是否正确"和"有没有高风险"。系统提供双模式输出：

- **摘要模式（默认）**：只展示关键推断摘要 + 确认按钮 + 高风险项警示。用户可一键切换详细模式。
  - 示例："推测修改文件：`src/pages/Login.tsx`（置信度：高）。i18n key `welcome_back` 已存在。无高风险项。推测是否有误？【正确 / 修正】"
- **详细模式**：展示完整补全包，包括依赖图、验收矩阵、执行 prompt envelope 等。适合复杂任务或需要精细审查时使用。

用户可在 `project-profile.yaml` 中设置默认模式：

```yaml
intake:
  default_mode: compact  # compact / verbose
```

### 0阶 Gateway 数据流与项目探测 SLA

0 阶项目探测的数据流：

```
外部请求 → orch_gateway.py (路由层)
         → gateway_intake.py (输入校验 + 标准化 → NormalizedIntent)
         → gateway_projection.py (状态投影 + 映射追踪 → ProjectedState)
         → gateway_evidence.py (证据收集 + 置信度标记 → EvidenceBundle)
         → 返回结构化输出
```

任一 helper module 加载失败时，Gateway 在 500ms 内降级到 `FALLBACK_HEURISTIC` 模式，记录降级事件到 `logs/gateway-fallback.jsonl`。

**项目探测 5 分钟 SLA**：

| 场景 | 行为 |
|------|------|
| 标准项目（含 package.json / pyproject.toml / Makefile） | 在 300 秒内输出完整《项目探测报告》 |
| 大型 monorepo（>1000 文件）或超时 | 输出 `status: partial` 的报告，至少包含技术栈，不阻塞后续流程 |
| 空目录（无构建文件） | 30 秒内输出 `status: unknown`，进入人工确认流程 |

降级路径保证：即使探测超时或失败，系统也能输出部分结果并继续工作，不会僵死或阻塞。

## 7. 信息保真机制

系统必须避免“模型补全以后，用户原意被覆盖”的问题。

每条关键信息都要保留：

- 来源：用户原话、代码事实、配置、测试结果、文档、模型推断。
- 置信度：高、中、低。
- 验证方式：已验证、待验证、不可验证、需用户确认。
- 影响范围：影响目标、方案、执行、测试、验收还是系统规则。

信息分层规则：

| 类型 | 是否可推进 | 说明 |
|------|------------|------|
| 用户原话 | 可作为最高优先级约束 | 除非与安全/法律/权限冲突 |
| 已验证事实 | 可作为执行和验收依据 | 来自代码、测试、配置、官方文档或 Gateway state |
| 系统补全 | 可推进，但要保留来源和置信度 | 高置信度默认自动推进 |
| 模型推断 | 只能作为候选判断 | 必须经证据或评审确认 |
| 低置信度内容 | 不可作为完成证据 | 只能作为 warning |
| 冲突信息 | 不可静默覆盖 | 进入 Conflict Ledger |

---

## 8. 一阶：方向辩论 → 对齐业务可行性

一阶不直接写代码，而是判断“这个方向是否应该做、怎么做才不偏”。

若用户请求包含客观错误（如“跳过测试直接上线”、“明文存密码”），一阶启动前先走**用户错误纠正机制**（见第 4 节），纠正完成后才进入辩论。

Kimi 基于《需求补全包》调度 Hermes 辩论团队：

- `dynamic_assembly`：自动选择需要哪些评审维度。
- `adversarial_debate`：方向争议大时做正反交锋。
- `jury_panel`：需要明确 verdict 时汇总投票。

输出《最佳选择报告》，至少回答：

- 方向是否可行。
- 业务价值和机会成本。
- 最大风险和阻塞点。
- 是否需要调整目标或非目标。
- 下一阶段应评审哪些技术路径。

如果方向报告高置信度、低风险、无冲突，系统自动进入二阶；否则才请求确认。

---

## 9. 二阶：方案辩论 → 锁定技术实现路径

二阶确定“怎么实现最稳”，并把需求变成可调度的 DAG。

需要补齐：

- 技术路线。
- 模块拆分。
- 文件/组件影响范围。
- 上下游依赖。
- 并行执行边界。
- 测试策略。
- 验收矩阵映射。
- 回滚或失败策略。

常用模式：

- `adversarial_debate`：路线争议大，充分暴露风险。
- `parallel_debate`：多维度独立评审。
- `risk_priority_matrix`：排序风险优先级。
- `cross_team_conflict_detector`：检测团队结论冲突。

说明：用户原表述中的 `riskprioritymatrix` 在 full registry 中的 canonical mode id 是 `risk_priority_matrix`。

输出《具体实现报告》，包含：

- 推荐方案和备选方案。
- 任务 DAG。
- 每个任务的输入、输出、写入范围和验收标准。
- 依赖与冲突处理策略。
- 需要上浮用户的风险项。

如果方案低风险、可逆、证据完整，系统自动进入三阶。

## 10. 三阶：具体执行 → 分层推进完成编码

三阶由 Hermes agents 管生命周期，Claude/Codex 做具体工程动作。

必须满足：

- 写入范围可控。
- 并行任务互不冲突，或有明确合并策略。
- 每个代理有独立 workspace 或等价隔离。
- 每个输出都有测试/审查/证据 artifact。
- Gateway 校验证据后才推进状态。
- **执行期间向用户推送进度摘要（执行心跳），避免长时静默造成焦虑。**

### 执行心跳机制

三阶执行可能持续数分钟（编码 → 测试 → 审查 → 复测）。完全静默会让用户产生"卡住了还是正常执行？"的焦虑。系统必须：

- **每 30 秒（或每完成一个子任务）推送一次结构化进度摘要**，包括：当前阶段、已完成任务数、正在进行的任务、预计剩余时间、是否被阻塞。
- **允许用户随时输入"状态？"或等效指令**，Gateway 返回当前 Run 的实时快照，不中断执行。
- **阻塞或异常时立即通知**，而非等到全部结束后才暴露。

**传输协议**：
- 心跳通过 **SSE（Server-Sent Events）** 推送到客户端，事件名 `orchestra.heartbeat`。
- 客户端断开 5 秒内重连可恢复，Gateway 缓存最近 3 条心跳。
- 若 SSE 不可用时（如终端环境），降级为**轮询**：客户端每 30 秒自动 `GET /orchestra/runs/{run_id}/projection`，Gateway 返回相同结构化数据。

**实时快照查询**：
- 用户输入 `"状态？"`、`"status"`、`"进度"` 或 `"?"` 时，Gateway 立即返回当前 Run 的只读快照。
- 快照内容包含：当前阶段、已完成/进行中/排队任务列表、最近 3 条事件、阻塞原因（如有）、预计剩余时间。
- **保证**：快照查询是只读操作，不会中断 worker 执行、不会触发新的状态变更。

示例进度摘要（SSE 推送）：

```text
[三阶执行中] 已完成 2/5 项任务
✅ rate_limiter.py 编码完成
✅ payment.py 修改完成
🔄 测试中（预计 30 秒）
⏳ 审查中（排队中）
```

Claude/Codex 分工：

- Codex：编码、修复、测试执行、结构化结果返回。
- Claude：代码审查、风险审计、专项评估、复审。
- 额外实例：特殊审计、交叉评审、隔离执行。

三层约束：

| 约束层 | 作用 |
|--------|------|
| Harness Engine | 物理阻断越权操作，例如敏感文件、生产写操作、危险命令 |
| AGENTS.md | 项目规范，例如编码风格、工具选择、目录边界 |
| SOUL.md | 决策偏好，例如保守实现、安全优先、简洁优先 |

输出《任务反馈文档》，记录修改、测试、审查、未解决问题和证据引用。

---

## 11. 四阶：改进实现 → 当场完成问题修复

四阶不是“评审后留待以后”，而是把问题立刻转成修复任务。

流程：

1. Kimi 读取《任务反馈文档》和验收矩阵。
2. 对问题归类：实现缺陷、测试不足、架构偏差、风险未处理、需求偏离。
3. 可自动修复的问题直接调度 Hermes。
4. 修复后重新测试、重新审查。
5. 超出原范围或触发风险边界时才上浮。

输出《改进报告》，记录问题、修复、复测证据和残余风险。

---

## 12. 五阶：全局评估 → 多维度验收

五阶检查“是否真的满足需求”，而不是只看代码能不能跑。

评估维度：

- 原始目标是否达成。
- 系统补全是否正确。
- 验收矩阵是否全部覆盖。
- 测试证据是否足够。
- 安全和合规是否过线。
- 性能和可靠性是否可接受。
- 文档和交接是否完整。
- 是否存在残余风险或未消解冲突。

推荐模式：

- `jury_panel`：形成总体 verdict。
- `meta_review`：审视前序评审是否遗漏。
- `cross_team_conflict_detector`：保留跨团队冲突。

如果 verdict 为 pass 且证据完整，系统可自动通过并通知用户；如果 fail/block，或 pass_with_warnings 但残余风险超阈值，才阻塞等待决策。

**残余风险阈值定义（用户可见）**：

| 风险等级 | 用户看到什么 | 系统行为 |
|----------|-------------|----------|
| 高 | "存在高风险残余项，必须确认后才能继续" | **阻塞**，必须用户或审批人显式决策（接受 / 回滚 / 重设计） |
| 中 | "存在 X 个中风险残余项" | 允许 ≤ 1 个；超过 1 个时**阻塞**等待决策 |
| 低 | "存在 X 个低风险建议关注项" | 允许 ≤ 3 个；超过 3 个时 verdict 仍为 `pass_with_warnings`，但通知中突出提示 |

**用户决策界面**：
当五阶因残余风险超阈值而阻塞时，系统向用户推送决策卡片：
```text
📋 五阶评估结果：pass_with_warnings（残余风险超阈值）
━━━━━━━━━━━━━━━━━━━━
阻塞原因：
• 中风险：Redis 超时 0.5s，极端网络抖动时可能误判（影响支付限流准确性）
• 中风险：新接口未做向后兼容，旧客户端可能解析失败

选项：
[接受风险并继续] → 进入六阶，残余风险记入审计日志
[回滚到三阶基线] → 丢弃本次变更，保留原代码
[升级到标准通道重审] → 重新执行完整六阶，补充测试与文档
━━━━━━━━━━━━━━━━━━━━
```

### pass_with_warnings 的主动摘要通知

五阶 verdict 为 `pass_with_warnings` 时，系统虽自动通过，但用户可能忽略残余风险。Gateway 必须在通知中显式摘要残余风险，确保用户知情权：

- 通知格式：
  ```text
  ✅ 任务已完成并自动通过。
  
  📋 残余风险（低）：Redis 超时 0.5s，极端网络抖动时可能误判。
  📋 建议关注：黑五期间留意 payment_rate_limited_total 指标。
  
  完整报告：[链接]
  ```
- 残余风险按等级排序：高/中/低。若存在中高风险，即使 verdict 为 pass_with_warnings 且阈值未超，也应在通知中突出显示。
- 用户可通过 `project-profile.yaml` 配置通知级别：
  ```yaml
  evaluation:
    warning_notification: summary  # none / summary / full
  ```

## 13. 六阶：持续改进 → 经验沉淀

六阶由 Kimi 做独立审计。

Kimi 读取：

- 原始需求和补全包。
- 依赖图和验收矩阵。
- 代理交互。
- 工具调用。
- 错误堆栈。
- 测试输出。
- 审查意见。
- Gateway 状态和审计记录。

Kimi 回答：

1. 产出是否符合预期？
2. 哪些信息补全是正确的，哪些补全造成误导？
3. 是否存在系统性错误、重复错误、低效循环或调度缺陷？
4. 哪些经验应该持久化到 AGENTS.md、SOUL.md 或配置策略，哪些应该丢弃？

经验沉淀必须保留适用边界，避免把偶然成功写成通用规则。

## 14. 用户实际使用方式

日常只需要这样说：

```text
目标：我要实现/修复/评估 X。
线索：相关文件、背景、接口、现象是 A、B、C。
约束：不要碰 D；如果需要 E，先问我。
验收：最终应满足 F、G、H。
```

### 快速通道示例

**用户**：登录页“欢迎使用”改成“欢迎回来”。

**Kimi**：检测到 P3 级变更（单文案修改）。规则引擎初筛通过，启动简单辩论确认……

> **frontend**（1 轮）：✅ 纯文案，无交互变更  
> **security**（1 轮）：✅ 无敏感信息
>
> 快速通道已确认，执行中……

**Kimi**（45 秒后）：已完成，变更在分支 `quickfix/copy-1855`，已自动合并至主干。

**Kimi**：✅ 已合并并归档。本任务无流程优化建议。

### 快速通道自动合并失败示例

**用户**：登录页"欢迎使用"改成"欢迎回来"。

**Kimi**：快速通道已确认，执行中……（45 秒后）

> ⚠️ 自动合并失败。
> 原因：目标分支 `main` 在您执行期间已有新提交，产生合并冲突。
> 分支：`quickfix/copy-1855`
>
> 可选操作：
> [1] **查看冲突并手动修复** — 系统展示冲突文件 diff
> [2] **升级到标准通道重审** — 重新执行完整测试与审查，由人工合并
> [3] **放弃本次变更** — 关闭 Run，保留原代码

**用户**（选择 1）：手动修复冲突。

**Kimi**：冲突已解决，重新尝试合并…… ✅ 已合并。

**说明**：自动合并失败时系统不会静默跳过，也不会强制覆盖他人提交。用户必须在 3 个选项中显式选择，Run 状态保持 `blocked` 直到用户决策。

### 新项目接入示例

**用户**（首次在该 repo 提需求）：给支付网关加限流。

**Kimi**：检测到这是新项目首次使用。正在自动探测项目信息……

> **项目探测报告（草稿）**
> - 语言：Python 3.12（来自 `pyproject.toml`）
> - 框架：FastAPI（来自依赖）
> - 测试：pytest（来自 `pytest.ini`）
> - 部署：Kubernetes（来自 `k8s/` 目录）
> - 发现风险标志：依赖中含 `stripe` → 标记为有支付流

**Kimi**：以上探测结果是否正确？请修正错误项（约 3-5 分钟）。

**用户**：正确。

**Kimi**：✅ 项目画像已生成，初始 AGENTS.md 和 SOUL.md 已创建。现在进入需求补全阶段。

如果用户没有写验收，Kimi 会先自动补出一版验收矩阵；只有无法可靠推断时才追问。

用户通常不需要：

- 手动调 Hermes Kanban。
- 指定每个 Claude/Codex 子任务。
- 手动拼接 Gateway 请求。
- 手动维护辩论团队组合。
- 检查每个中间文件。

用户需要确认的例外：

- L3/L4 高风险事项。
- 生产发布、权限、密钥、CI/CD、根规则。
- 目标偏离。
- 无法自动消解的冲突。
- 关键判断低置信度。
- protected target 的经验沉淀或配置修改。

## 15. 辩论团队扩展规范

### 核心团队（16 支，不可删除）

```text
security, compliance, data_engineering, devops_sre, frontend,
ai_feature, scalability_arch, chaos_engineering, platform,
```

### 扩展团队（项目级自定义）

```json
{
  "custom_teams": [
    {
      "id": "payment_compliance",
      "extends": "compliance",

```

**扩展规则**：
1. 必须声明 `extends`，继承核心团队基线行为。
2. `id` 不能与核心团队冲突。
3. `prompt_injection` 不得覆盖核心安全红线。
4. 多项目复用的自定义团队可提升到组织级。

| 数据平台 | data_eng, platform, scalability, observability, security | frontend, business | data_quality, lineage |

## 16. 项目接入配置：混合模式

### 流程

1. **Kimi 自动探测**（30 秒）：读取文件树、依赖文件、CI/CD 配置、测试目录结构。
2. **展示探测报告草稿**，请求开发者确认或修正。
3. **开发者确认**（3-5 分钟，8-12 个问题）。
4. **生成项目画像**：`.hermes/project-profile.yaml`。
5. **生成初始规则文件**：`AGENTS.md`（基于模板 + 探测结果）、`SOUL.md`（极简，待沉淀）。
6. **接入完成**，进入正常工作流。

### 探测规则

| 探测项 | 信号来源 | 置信度 |
|--------|---------|--------|
| 编程语言 | `package.json` / `pyproject.toml` / `go.mod` / `Cargo.toml` | 高 |
| Web 框架 | 依赖文件中的框架标识 | 高 |
| 测试框架 | 文件树 `tests/`、`pytest.ini`、`jest.config` 等 | 中高 |
| 数据库 | `docker-compose.yml`、ORM 依赖 | 中 |
| 部署目标 | `.github/workflows/`、`Dockerfile`、`k8s/`、`vercel.json` | 中 |
| 支付流 | 代码/依赖中含 `stripe`、`paddle`、`braintree` | 高 |
| PII | 代码中含 `email`、`phone`、`gdpr`、`ccpa` | 中 |
| 合规 | 文件/文档中含 `SOC2`、`HIPAA`、`PCI` | 中 |

### project-profile.yaml 核心字段

```yaml
profile:
  tech_stack:
    language: python
    framework: fastapi
    test_runner: pytest
  deploy:
    target: kubernetes
  risk_flags:
    has_payment_flow: true
    has_pii_data: true
  protected_targets:
    - pattern: "k8s/production/.*"
      level: L4
  evidence_requirements:
    test: { command: "pytest --cov", min_coverage: 80 }
  debate:
    inactive_teams: [frontend]
    custom_teams: []
  quick_channel:
    rollout_phase: observe_only   # observe_only / calibrating / enabled
    auto_merge: true
    notification: compact         # silent / compact / verbose
  evaluation:
    warning_notification: summary # none / summary / full
```

**Rollout Phase 说明**：
- `observe_only`（默认）：所有任务实际走标准通道，但系统会标注"本可走快速通道"，用于收集校准数据。新项目或重大配置变更后强制回到此阶段。
- `calibrating`：允许走快速通道，但需要人工二次确认（不自动合并），用于验证规则准确性。
- `enabled`：完全启用快速通道，满足条件时自动合并。
- 切换条件由系统自动判定（基于误判率），但用户可手动申请提前切换，需 L3 审批。

## 17. 与当前仓库实现的对应关系

固定阶段报告对应：
