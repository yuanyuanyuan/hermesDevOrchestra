# TDD 长任务工作流操作手册

> **定位**：原 [tdd skill](/home/stark/.claude/skills/tdd/SKILL.md) 在长任务场景下的**前置对齐与协作补充**。原 skill 的 RED/GREEN/REFACTOR、Vertical Slicing、行为测试等核心在长任务中绝对主导，本手册只解决多模块并行时的边界对齐问题。
> 
> **状态**：社区草案。流程整合部分未经大规模生产验证，建议小规模试点。

## 适用场景

| 场景 | 判定标准 | 是否适用 |
|------|---------|---------|
| 短任务 | < 1 天，单模块 | ❌ 直接用原 skill |
| 中任务 | 1-3 天，2-3 个模块 | ⚠️ 选择性使用 |
| **长任务** | **> 3 天，多模块，接口需演化** | **✅ 完整使用** |

## 核心原则（与原 Skill 一致）

以下原则在长任务中**不可动摇**：
- 测试验证行为，不验证实现
- Vertical Slicing：一个测试 → 一个实现 → 重复
- 契约层测试（进入仓库的测试）必须走公共接口
- 不在 RED 状态下重构
- 一次只关注一个可验证行为
- 每个单测试循环必须通过原 skill 的 Checklist

**补充原则**（长任务场景）：
- 骨架先行：先对齐模块边界，再进入 TDD 循环
- 渐进接口：接口在演化中逐步稳定
- 时间盒切片：以半天纯开发时间为约束，非固定行为数量

## 概念出处

| 概念 | 原始出处 | 文档用法 |
|------|---------|---------|

## 关键区分

**Walking Skeleton vs Spike**

| | Walking Skeleton | Spike |
|--|-----------------|-------|
| 目的 | 验证架构可行性和模块连通 | 降低技术风险 |
| 产出 | **可演进基线**（保留到生产） | 通常丢弃 |
| 内容 | 真实依赖注入结构 + 硬编码业务逻辑 | 可工作原型，质量不重要 |
| 时间 | 1-2 天 | 数小时到 1 天 |

## Phase 0: Walking Skeleton（1-2 天）

### 目标
建立**可运行骨架**，验证模块边界和技术栈连通。**接口契约在此阶段不冻结**。

### 步骤
1. **模块划分**：按业务能力和变更频率划分，画出依赖图，识别叶子模块（优先）和入口模块（最后）
2. **建立骨架**：每个模块建立真实的依赖注入/适配器结构，业务逻辑可硬编码，但**模块边界和接口签名必须真实**
3. **编写骨架测试**：一个端到端测试，验证流程跑通无异常
4. **验证骨架**：测试通过后才进入 Phase 1

### 接口冻结规则
- Phase 0：接口可变更
- Phase 1：通过 Tracer Bullet 演化
- Mid-way 检查点后：实验性接口冻结为稳定接口

### 安全边界预设
若系统有安全要求，入口模块的认证/授权/输入验证接口应在 Phase 0 定义 Stub，避免叶子优先执行时安全 batch 被拖到最后。

## Phase 1: 模块级 Vertical Slicing

### 执行顺序
按依赖拓扑从叶子向入口推进。**无依赖模块可并行**。

### Step 1: Planning（30 分钟）
- 列出本模块核心行为（按优先级）
- 定义内部约定接口（命名前缀 `_internal_`）
- 识别依赖边界行为
- 确认测试数据 Factory 的初始化方式

### Step 2: Tracer Bullet（1-2 小时）
写**一个**测试验证最核心行为：
- 不依赖外部模块 → 直接测试真实实现
- 依赖外部模块 → 使用 **Fake**（内存实现），禁止 Mock 验证框架

**Fake 警示**：若 Fake 需要分支逻辑或状态管理，视为**耦合信号**——优先简化接口或优先实现该依赖模块。

### Step 3: Mini-Batch（半天时间盒）
按**内聚性**分组，每组为一个 batch：
- **时间盒**：纯开发时间 4 小时内（复杂领域可放宽至 1 天）
- **数量**：通常 2-6 个内聚行为，但**以内聚性和时间盒为约束**
- **内容**：聚焦同一业务场景（如"支付成功流程"）

Batch 内部严格 RED/GREEN/REFACTOR 单循环。Batch 结束后模块内重构。

**Batch 完成验证**：模块级契约测试 + 当前模块与**已完成模块**的集成测试全绿。端到端骨架测试仅在**模块入列**时执行。

### Step 4: 接口稳定化
模块完成后：
- 审视内部接口，稳定后提升为实验性接口（`_experimental_` 前缀）
- 评估结构层测试（见下文）
- 保留契约层测试
- 替换骨架中的硬编码实现
- 运行端到端骨架测试验证系统连通

### 模块完成标准
- [ ] 核心业务能力（happy path + 主要异常）已覆盖
- [ ] 若模块是入口/边界层：安全边界（权限、输入验证）已覆盖
- [ ] 错误处理策略（降级、重试、超时）已定义
- [ ] 接口文档已更新

**不强制**（可延后）：性能基准、日志/监控、配置化、国际化。

## 双层测试体系

### 契约层（Contract Tests）——仓库资产
- **目标**：验证可观察行为
- **对象**：模块公共接口、系统入口
- **生命周期**：永久资产
- **规则**：禁止 mock 内部协作者；断言聚焦业务结果

### 结构层（Structural Tests）——个人辅助
> ⚠️ **重要**：结构层测试**不是 TDD 正式产出**。它是工程师个人验证内部算法的临时工具，**不进入代码仓库**。

- **目标**：个人快速验证内部算法
- **对象**：纯函数、算法、边界条件
- **生命周期**：临时草稿，模块完成后删除
- **规则**：允许针对具体函数；失败不代表功能失败，契约层才是唯一标准

**处置规则**：
- 契约层通过 + 结构层通过 → 结构层使命完成，删除
- 契约层通过 + 结构层失败 → 忽略，继续开发
- 契约层失败 → 修复代码，结构层仅用于快速定位

**执行策略**：
- 日常循环：结构层 + 当前模块契约层
- Batch Checkpoint：模块全量契约层
- 模块完成：模块契约层 + 端到端骨架测试
- 提交前 / CI：全量契约层

## 渐进接口暴露

| 级别 | 命名（动态语言） | 阶段 | 稳定性 |
|------|----------------|------|--------|
| 内部约定 | `_internal_` 前缀 | Phase 1 开发期 | 允许频繁变更 |
| 实验性接口 | `_experimental_` 前缀 | 模块完成后 | 调用方容忍变更 |
| 稳定接口 | 无前缀 | Mid-way 后 | 变更需向后兼容 |

**升级**：内部 →（模块测试全绿）→ 实验性 →（Mid-way 通过）→ 稳定

**变更规则**：稳定接口原则上不变更。若必须变更：提供向后兼容过渡方案 + 技术负责人审批。

## 检查点与回滚

### 检查点

| 检查点 | 触发时机 | 内容 | 失败处理 |
|-------|---------|------|---------|
| 骨架检查 | Phase 0 结束 | 端到端流程跑通 | 阻塞，不进入 Phase 1 |
| 模块入列 | 每完成一个模块 | 替换骨架实现，跑端到端测试 | 按决策流程处理 |
| Mid-way | 关键路径完成 **或 Sprint 第 7 天**（先到为准） | 全量契约 + 端到端测试 | 接口评审，调整契约 |
| 最终集成 | 所有模块完成 | 全量契约 + 端到端测试 | 按优先级修复 |

### 检查点失败决策流程

```
失败
├─ 涉及接口语义变更？ → 回退到 Phase 1，重新设计接口
├─ 涉及模块职责/数据流调整？ → 架构评审，调整骨架
└─ 局部实现偏差？ → 就地修复 + 补契约层测试
```

### 回滚策略

| 场景 | 操作 |
|------|------|
| 单个模块入列失败 | 回退代码到入列前版本，恢复骨架硬编码 |
| 多模块交叉失败 | 按依赖拓扑**逆序**回滚 |
| 涉及 Schema 变更 | 代码回滚优先；Schema 通过迁移工具管理，采用前向兼容策略 |

## 多人并行开发

### 协调机制
- **接口变更广播**：实验性接口变更时，变更方在团队频道/文档广播；依赖方负责同步更新 Fake
- **Fake 共享**：单体仓库 → 共享 Fake 目录；多仓库 → 使用 OpenAPI/Protobuf 契约文件生成 Fake
- **代码审查**：模块入列前同模块审查；Mid-way 前跨模块接口审查
- **并行冲突**：相互依赖模块并行开发时，先定义共享接口契约文档，双方基于契约各自开发

### 最小角色
- **技术负责人**：Phase 0 和 Mid-way 评审
- **模块负责人**：模块内 batch 规划、接口广播
- **集成协调员**：端到端测试和检查点执行

---

## 需求变更处理

| 变更层级 | 判定标准 | 决策人 | 处理方式 |
|---------|---------|--------|---------|
| 模块内 | 不影响其他模块接口 | 开发者自判 | 按 Mini-Batch 处理 |
| 跨模块 | 影响实验性接口 | 技术负责人 | 修改并通知依赖方 |
| 架构级 | 新增/删除模块 | 架构评审会 | 回到 Phase 0 |

> **判定原则**：若变更导致其他模块契约层测试失败，即为跨模块或以上层级。

**契约层测试**：随接口变更同步更新，删除旧接口绑定测试或重构为兼容性回归测试。

## 遗留系统集成

### 防腐层（Anti-Corruption Layer）
在遗留系统边界建立独立适配器模块，将遗留接口翻译为系统内部标准接口。**绝不**让遗留数据结构泄露到系统内部。

### 特征化测试（Characterization Test）
修改遗留接口前，捕获其当前实际行为：
1. 通过日志/抓包记录遗留系统对典型输入的输出
2. 编写测试复现相同输入输出
3. 标记为 `Characterization` 以便区分
4. 修改前后测试必须保持通过（不验证正确性，只验证不变性）

## 测试数据管理

- **跨模块对齐**：建立共享测试数据目录，定义标准测试实体
- **Fixtures 共享**：契约层用共享 Factory；结构层允许简化 Fixture
- **数据库迁移**：Schema 变更与代码同一 PR 提交；提交前验证迁移可逆性

## 安全与可观测性

### 安全测试
- 入口/边界层模块的输入验证、权限检查在 Phase 1 中作为**优先 batch**
- 契约层覆盖：越权失败、非法输入拒绝、敏感数据不泄露

### 可观测性
- **日志**：模块完成时，核心路径记录 INFO 级别日志
- **监控**：Mid-way 后核心接口暴露调用次数、错误率、延迟
- **追踪**：最终集成时验证跨模块请求链的追踪 ID 传递

### 错误处理
Planning 阶段定义：降级策略、重试退避、熔断机制（至少以 TODO 记录）。

## 常见陷阱

| 陷阱 | 对策 |
|------|------|
| 骨架太薄 | 至少定义核心 DTO 结构 |
| 骨架太厚 | 只保证连通，不处理业务逻辑 |
| Fake 泛滥 | Fake 需分支逻辑时视为耦合信号，优先简化接口 |
| 结构层提交到仓库 | 结构层是个人辅助，提交前删除 |
| Mini-Batch 过大 | 强制时间盒约束（半天），超时拆分 |
| 接口过早稳定 | Mid-way 前不冻结 |
| 需求变更无门 | 按模块内/跨模块/架构级分级处理 |
| 遗留系统污染 | 强制使用防腐层 |

## 术语表

| 术语 | 定义 |
|------|------|
| Walking Skeleton | 可运行骨架，真实依赖注入 + 硬编码业务逻辑 |
| Spike | 实验性原型，产出通常丢弃 |
| Tracer Bullet | 模块级首个连通性验证测试 |
| Mini-Batch | 时间盒约束的内聚行为组，内部严格 Vertical Slicing |
| Fake | 有业务语义的简化实现 |
| Stub | 固定返回值的简单替身 |
| 契约层测试 | 验证可观察行为的永久资产，遵循原 skill checklist |
| 结构层测试 | 个人开发辅助，不进入仓库 |
| Factory | 动态生成测试实体的工厂函数 |
| Fixture | 预定义的固定测试数据 |
| 防腐层 | 遗留系统边界的适配器模块 |
| 特征化测试 | 捕获遗留系统当前行为，不验证正确性 |
