# SP 估算参考（增强版 v2）

## 产能基准

| 指标 | 值 |
|------|-----|
| 有效编码时间/天 | 6 小时 |
| 单 Sprint 有效工时 | 30 小时 |
| **Sprint 容量** | **7 SP** |
| **最大单任务容量** | **8 SP（超过必须拆分）** |

## Fibonacci 估算尺

| SP | 预估工时 | 典型任务 |
|----|----------|----------|
| 1 | ~4h | 配置修改、简单 bug 修复、文档更新、单字段 schema 变更 |
| 2 | ~8h（1 天） | 单个接口、简单组件、单元测试补充、别名映射表 |
| 3 | ~12h | 模块级功能、中等复杂度集成、状态机单状态实现 |
| 5 | ~20h | 跨模块功能、复杂业务逻辑、多文件协调重构 |
| 8 | ~32h | 架构级变更、多模块联动、安全门控完整实现 |
| 13 | ~52h | **必须拆分**，不允许作为单个任务 |

---

## 估算原则（新增）

1. **包含开发 + 单元测试 + 集成测试 + Code Review 修改 + 文档同步时间**
2. **不确定的任务取高估值**
3. **含外部依赖的任务 +1 SP 缓冲**
4. **超过 8 SP 的单元必须拆分后再分配**
5. **每个 Test Scenario 至少分配 0.5 SP 的测试实现成本**
6. **涉及负向测试/异常路径/边界条件的任务 +1 SP**
7. **估算结果必须进行"均一化检查"——不允许所有 Sprint 容量利用率相同**

---

## 基础估算（快速判断规则）

| 场景 | 基础 SP |
|------|---------|
| 改动 ≤3 个文件，逻辑简单 | 1-2 SP |
| 改动 4-6 个文件，有业务逻辑 | 3 SP |
| 改动 7-10 个文件，跨模块 | 5 SP |
| 改动 >10 个文件，或涉及架构 | 8 SP（考虑拆分） |

## 估算因子加成体系（评审经验沉淀）

在基础估算之上，按以下因子逐项累加：

### F1 — 架构债务系数

| 条件 | 加成 |
|------|------|
| 需要向已有单文件（>3000 行）新增逻辑 | +2 SP |
| 需要抽取 helper module / seam（但无先例） | +2 SP |
| 涉及循环依赖解除或共享核心模块抽取 | +1 SP |
| 需要保持与 legacy/MVP 路径的双轨兼容 | +1 SP |

> **红线检查**：如果某 Sprint 的唯一代码修改文件是单个大文件（如 `orch_gateway.py`），必须触发架构债务审查，考虑拆分 seam extraction 为独立子任务。

### F2 — 安全/权限系数

| 条件 | 加成 |
|------|------|
| 涉及身份认证机制设计（token/mTLS/签名） | +2 SP |
| 涉及权限矩阵/ACL/授权决策 | +1 SP |
| 涉及安全红线定义或 prompt injection 检测 | +2 SP |
| 涉及 protected target 审批流程 | +1 SP |
| 涉及数据防篡改/审计链/密码学签名 | +2 SP |

### F3 — Schema 三重同步系数

| 条件 | 加成 |
|------|------|
| 需要同时修改 schema.md + schema.json + 代码实现 | +2 SP |
| 需要新增 artifact/数据模型定义 | +1 SP |
| 涉及向后兼容的字段变更 | +1 SP |
| 需要编写 schema↔实现一致性校验脚本 | +1 SP |

### F4 — 测试覆盖系数

| 条件 | 加成 |
|------|------|
| 每个 Test Scenario 的实现成本 | +0.5 SP |
| 需要新增专用测试脚本（非复用已有） | +1 SP/脚本 |
| 涉及负向测试/异常路径/边界条件 | +1 SP |
| 涉及 E2E / 闭环回归测试 | +3 SP |
| 需要 mock 基础设施/外部服务 | +1 SP |

### F5 — 文档/配置同步系数

| 条件 | 加成 |
|------|------|
| 需要同步更新 ≥3 份文档 | +1 SP |
| 涉及 ADR（架构决策记录）新增/修改 | +0.5 SP |
| 涉及用户流程指南变更 | +0.5 SP |
| 涉及配置格式变更 + 向后兼容 | +1 SP |

### F6 — 跨 Sprint 接口契约系数

| 条件 | 加成 |
|------|------|
| 输出作为下游 Sprint 的输入（需定义契约） | +1 SP |
| 需要消费上游 Sprint 的输出（契约尚未稳定） | +1 SP |
| 涉及状态机/阶段流转的跨 Sprint 衔接 | +1 SP |

### F7 — 未知/模糊系数

| 条件 | 加成 |
|------|------|
| 关键技术方案尚未定义（算法/协议/格式） | +2 SP |
| 依赖外部团队/第三方决策 | +1 SP |
| 验收标准中存在"完整""及时""足够"等不可量化词 | +1 SP |

## 依赖加成（保留）

| 依赖类型 | 额外 SP |
|----------|---------|
| 无依赖 | 0 |
| 内部模块依赖 | 0 |
| 外部 API 依赖 | +1 |
| 第三方服务依赖 | +1 |
| 需要基础设施变更 | +2 |

---

## 估算复核清单（生成前必须检查）

```markdown
- [ ] 该 Sprint 是否涉及向单一大文件堆叠逻辑？如是，是否已拆分 seam extraction？
- [ ] 安全/权限相关任务是否已加安全系数？
- [ ] Schema 变更是否计算了三重同步成本？
- [ ] 每个 Test Scenario 是否都分配了测试实现 SP？
- [ ] 是否有负向测试/异常路径？如是，是否已加成？
- [ ] 关键技术方案（算法/协议/格式）是否已定义？如未定义，是否已加未知系数？
- [ ] 验收标准是否可量化？是否存在"完整""及时""足够"等模糊词？
- [ ] 该 Sprint 的容量利用率是否与相邻 Sprint 不同？（均一化是危险信号）
- [ ] 是否定义了向下游 Sprint 输出的接口契约？
- [ ] 是否存在前置依赖未完成的依赖风险？
```
