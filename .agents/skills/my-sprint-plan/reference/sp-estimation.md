# SP 估算参考

## 产能基准

| 指标 | 值 |
|------|------|
| 有效编码时间/天 | 6 小时 |
| 单 Sprint 有效工时 | 30 小时 |
| **Sprint 容量** | **7 SP** |

## Fibonacci 估算尺

| SP | 预估工时 | 典型任务 |
|----|----------|----------|
| 1 | ~4h | 配置修改、简单 bug 修复、文档更新 |
| 2 | ~8h（1 天） | 单个接口、简单组件、单元测试补充 |
| 3 | ~12h | 模块级功能、中等复杂度集成 |
| 5 | ~20h | 跨模块功能、复杂业务逻辑 |
| 8 | ~32h | 架构级变更、多模块联动 |
| 13 | ~52h | **必须拆分**，不允许作为单个任务 |

## 估算原则

1. **包含开发 + 单元测试 + Code Review 修改时间**
2. **不确定的任务取高估值**
3. **含外部依赖的任务 +1 SP 缓冲**
4. **超过 8 SP 的单元必须拆分后再分配**

## 快速判断规则

- 改动 ≤3 个文件，逻辑简单 → 1-2 SP
- 改动 ≤5 个文件，有业务逻辑 → 3 SP
- 跨模块改动，有集成 → 5 SP
- 架构变更或新技术引入 → 8 SP

## 依赖加成

| 依赖类型 | 额外 SP |
|----------|---------|
| 无依赖 | 0 |
| 内部模块依赖 | 0 |
| 外部 API 依赖 | +1 |
| 第三方服务依赖 | +1 |
| 需要基础设施变更 | +2 |
