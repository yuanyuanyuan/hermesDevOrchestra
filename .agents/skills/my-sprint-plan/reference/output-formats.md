# 输出格式参考（增强版 v2）

---

## Spec 格式（功能规格说明）

与 v1 基本一致，增加以下章节：

```markdown
# [项目名] 功能规格说明

## 概述
[1-2 句话描述功能目标和范围]

## 功能需求

### FR-1: [需求名称]
- **描述**: [详细描述]
- **验收标准**: [可验证的条件，必须量化]
- **优先级**: [P0/P1/P2]

## 非功能需求
- **性能**: [具体指标，含时间阈值]
- **安全**: [具体要求，含 enforce 机制]
- **兼容性**: [具体约束，含向后兼容策略]

## 接口契约
### API 端点
| 方法 | 路径 | 描述 | 请求 schema | 响应 schema | 错误码 |
|------|------|------|-------------|-------------|--------|

### 数据模型
| 实体 | 字段 | 类型 | 约束 | 默认值 | 变更版本 |
|------|------|------|------|--------|----------|

## 范围边界
### 包含
- [功能点]

### 不包含
- [明确排除的功能点]

## 风险与依赖
| 风险 | 影响 | 缓解措施 | 监控指标 |
|------|------|----------|----------|

## 成功指标（新增，对应 PRD 第 11 章）
| 指标 ID | 事件源 | 聚合规则 | 阈值 | 验证方式 |
|---------|--------|----------|------|----------|
```

## Schema 格式（API/数据库变更）

与 v1 基本一致，增加以下要求：

```markdown
# [项目名] Schema 变更说明

## API 变更
### 新增/修改端点
[含请求体/响应/错误码，与 v1 相同]

## 数据模型
[含字段/类型/约束，**新增：变更版本号、向后兼容标记**]

## 持久化说明（新增）
- 存储介质：[文件/数据库/缓存]
- 原子写策略：[write-to-temp + rename / 事务 / 无]
- 并发控制：[文件锁/写时复制/版本号/无]
- 损坏检测与恢复：[校验和/备份/重放/无]

## Schema 一致性校验（新增）
- [ ] schema.md ↔ schema.json 字段名一致
- [ ] schema.json ↔ 代码实现校验逻辑一致
- [ ] 自动化校验脚本已编写并加入 CI
```

## Plan-Sprint 格式（增强版）

**每个 Sprint Plan 必须包含以下章节，不允许压缩为 19 行任务清单：**

```markdown
# Sprint {N} Plan

## 元信息
- **总故事点**: {sp} SP / 7 SP 容量
- **任务数**: {count} 项
- **前置依赖状态**: [上游 Sprint 完成状态，如 U{N-1} 已完成且 helper modules 就绪]
- **向下游输出契约**: [本 Sprint 输出什么 artifact/接口/数据格式供下游消费]

## 任务清单
| # | U-ID | 任务 | SP | 依赖 | 状态 |
|---|------|------|----|------|------|

## 接口契约变更（新增）
| 契约项 | 变更类型 | 上游 Sprint | 下游 Sprint | 详细说明 |
|--------|----------|-------------|-------------|----------|
| {artifact_name} | 新增/修改 | U{N-1} | U{N+1} | {描述} |

## 详细说明

### Task 1 ({uid}): {name}

#### 目标
{goal}

#### Approach（必须保留 source-plan 的 Approach 细节）
{逐条 approach 项，不丢失设计要点}

#### 验收标准（必须量化，逐条独立）
- [ ] {可量化的验收条件 1}
- [ ] {可量化的验收条件 2}

> ⚠️ **模糊词检测**：验收标准中不得出现以下词汇（除非已量化定义）：
> "完整"、"及时"、"足够"、"合理"、"适当"、"有效"、"正确"、"正常"、"尽快"

#### 测试矩阵（Test Scenario → 测试方法映射）
| # | Test Scenario | 验证方式 | 预期结果 | 测试脚本 |
|---|---------------|----------|----------|----------|
| 1 | {scenario} | {单元/E2E/集成} | {断言} | {脚本名} |

#### 负向测试/边界条件（新增）
- [ ] {负向用例 1：什么条件下应该阻塞/失败}
- [ ] {负向用例 2：什么条件下应该降级/回退}

#### 涉及文件
{Create/Modify/Delete 清单，每项说明具体变更范围}

#### 风险与降级策略
| 风险 | 缓解措施 | 降级方案 |
|------|----------|----------|
| {风险} | {措施} | {若失败则...} |
```

## Checklist-Sprint 格式（增强版 — 四级结构）

**Checklist 不再是 Plan 验收标准的"缩水复制版"，必须包含以下四个独立区块：**

```markdown
# Sprint {N} 验收清单

## 一、架构红线合规（新增，不通过则整体阻断）
- [ ] 新增逻辑优先落入 helper modules，非直接堆叠单一大文件
  - 验证：`orch_gateway.py` 新增代码行占比 < 20%
- [ ] 若必须修改单一大文件，已抽取独立 helper 并附带单元测试
- [ ] 接口契约变更已同步到 schema.md / schema.json
- [ ] 自动化校验脚本通过（如有 schema/契约变更）

## 二、功能验收（每条验收标准独立可勾选）
> 来源：将 Plan 中的验收标准逐条拆分为独立 checkbox，**不允许用分号拼接多条**

- [ ] **U{N}.1** — {量化验收条件 1}
  - 验证方式：{具体命令或断言}
  - 通过标准：{exit 0 / 字段非空 / 返回值匹配}
- [ ] **U{N}.2** — {量化验收条件 2}
  - 验证方式：...
  - 通过标准：...

## 三、测试覆盖（新增，必须映射 Test Scenarios）

### 3.1 正向测试（Test Scenarios）
| # | Scenario | 测试脚本 | 状态 |
|---|----------|----------|------|
| 1 | {scenario 1} | {脚本名} | ⬜ |
| 2 | {scenario 2} | {脚本名} | ⬜ |

### 3.2 负向测试/边界条件
- [ ] {负向用例 1：阻塞条件测试}
- [ ] {负向用例 2：降级/回退测试}
- [ ] {负向用例 3：异常输入处理}

### 3.3 回归测试范围
- [ ] 本 Sprint 相关测试套件全部 exit 0
- [ ] 非本 Sprint 测试失败不阻塞交付，但需确认不影响 main 稳定性

## 四、文档/Schema/配置同步（新增）
- [ ] 所有涉及文件变更已逐项确认（Create/Modify/Verify）
- [ ] schema.md 与 schema.json 一致（差异 0 项）
- [ ] 用户流程文档已同步更新
- [ ] ADR/架构文档已同步更新（如有架构变更）
- [ ] 配置变更已验证向后兼容

## 验证命令
```bash
# 必须明确列出每个命令的通过标准
{命令 1}  # 期望：exit 0，输出包含 "..."
{命令 2}  # 期望：exit 0，输出不包含 "ERROR"
```

## 签核
- [ ] 开发完成
- [ ] 全部测试通过（含负向测试）
- [ ] Code Review 完成
- [ ] 文档与实现一致性已确认
- [ ] 架构红线合规已确认
- [ ] 合并到 main
```

## 可量化验收标准写作指南（新增）

### ❌ 不合格示例
- "输出完整的探测报告" → "完整"未定义
- "系统及时响应" → "及时"未量化
- "保证足够的测试覆盖" → "足够"不可验证
- "合理处理异常情况" → "合理"主观

### ✅ 合格示例
- "探测报告包含技术栈/测试命令/部署目标/风险标志 4 个字段，缺失任一项返回 422"
- "API 响应 P99 延迟 < 200ms"
- "单元测试行覆盖率 ≥ 80%，且核心路径分支覆盖率 100%"
- "worker 崩溃后 Gateway 在 90 秒内标记 session 为 timed_out"

### 模糊词检测清单（生成时必须扫描）

```
完整、及时、足够、合理、适当、有效、正确、正常、尽快、
适当、充分、必要、主要、关键、重要、良好、优化、完善、
确保、保证、尽量、力争、原则上、一般情况下、视情况而定
```

> 如果验收标准中包含以上词汇且无具体数值/阈值/判定算法，
> **必须在生成报告中标记为"不可量化"，并要求补充量化定义。**

## 跨 Sprint 接口契约格式（新增）

在 `sprint-overview.md` 中增加以下章节：

```markdown
## 跨 Sprint 接口契约

| 契约名称 | 输出方 | 输入方 | 数据格式 | 状态 |
|----------|--------|--------|----------|------|
| intake_package | U2 | U3/U4/U5 | JSON (schema v1) | 已定义 |
| direction_debate_ticket | U5 | U6 | JSON (schema v1) | 待定义 |
| global_evaluation_report | U11 | U12 | JSON (schema v1) | 待定义 |

## 已知限制与债务

| Sprint | 已知限制 | 承接 Sprint | 缓解措施 |
|--------|----------|-------------|----------|
| S5 | alias-mapping.json 格式未定义 | S5 内必须完成 | 阻塞项 |
| S9 | merge strategy 未定义 | S9 或 S10 | 临时用 abort-on-conflict |
```
