# PR Review Comment 格式规范

> 基于 GitHub PR Review API / Review Comment API / Issue Comment API 文档设计。
> 目标：同时服务人类可读性和 Agent 可解析性。

## 双层格式设计

所有 review 相关 comment 采用**双层格式**：

- **外层（可视化层）**：Markdown — 供人类阅读，含 emoji、表格、折叠区域、代码块
- **内层（元数据层）**：HTML Comment — 供 Agent 解析，含结构化 JSON

```markdown
<!--AGENT_META:{"type":"pr_review","version":"1.0",...}-->

# 人类可读标题
...

<!--AGENT_FINDING:{"id":"f-1","level":"P0",...}-->
```

> **为什么用 HTML Comment？**
> - GitHub Markdown 渲染器会忽略 `<!-- -->` 内容，不在 UI 中显示
> - API 返回的原始 `body` 字段完整保留注释内容
> - Agent 可通过正则 `/<!--AGENT_(META|FINDING):({.*?})-->/g` 提取

---

## 模板类型总览

| 模板 | 用途 | API |
|------|------|-----|
| `pr_review_body` | Reviewer 的整体 review 报告 | `create_pull_request_review` body |
| `pr_review_inline` | 行级 inline comment | `create_pull_request_review` comments[] |
| `pr_review_body_fallback` | Body-only fallback（inline API 失败时） | `create_pull_request_review` body only |

## 模板 1: Review Body（整体报告）

### AGENT_META Schema

```json
{
  "type": "pr_review",
  "version": "1.0",
  "pr_number": 42,
  "reviewer": "stark-008",
  "commit": "abc123def...",
  "review_type": "initial | re_review",
  "check_items": 15,
  "pass_count": 12,
  "fail_count": 3,
  "na_count": 0,
  "conclusion": "PASS | HAS_BLOCKERS",
  "findings": [
    {
      "id": "f-1",
      "level": "P0 | P1 | P2 | P3",
      "file": "src/pool.py",
      "line": 45,
      "title": "Memory leak in connection pooling",
      "category": "bug | style | architecture | doc | test | security | performance"
    }
  ]
}
```

### 完整模板

```markdown
<!--AGENT_META:{"type":"pr_review","version":"1.0","pr_number":${PR_NUMBER},"reviewer":"stark-008","commit":"${HEAD_OID}","review_type":"${REVIEW_TYPE}","check_items":${CHECK_ITEMS},"pass_count":${PASS},"fail_count":${FAIL},"na_count":${NA},"conclusion":"${CONCLUSION}","findings":[${FINDINGS_JSON_ARRAY}]}-->

# 🔍 PR Review Report — PR #${PR_NUMBER}

**Reviewer:** stark-008 | **Commit:** \`${HEAD_OID}\` | **Type:** ${REVIEW_TYPE_LABEL}

## 📊 Summary

| Metric | Value |
|--------|-------|
| Check Items | ${CHECK_ITEMS} |
| ✅ PASS | ${PASS} |
| ❌ FAIL | ${FAIL} |
| ⏸️ N/A | ${NA} |
| **Conclusion** | **${CONCLUSION_EMOJI} ${CONCLUSION}** |

---

${FINDINGS_SECTION}

---

> _This review was generated by ce-code-review. For questions, tag @stark-008._
```

### Findings 按级别分组渲染

**P0 区块示例：**

```markdown
## ❌ P0 — Critical Issues

<details open>
<summary><b>[P0] ${TITLE} — <code>${FILE}:${LINE}</code></b></summary>

**Description:** ${DESCRIPTION}

**Evidence:**
```
${EVIDENCE_CODE}
```

**Suggestion:** ${SUGGESTION}
</details>

<!--AGENT_FINDING:{"id":"${ID}","level":"P0","file":"${FILE}","line":${LINE},"title":"${TITLE}","category":"${CATEGORY}"}-->
```

**P1/P2/P3 区块类似，emoji 分别为 ⚠️ / 💡 / 📝。**

### 复核场景额外区块

```markdown
## 🔄 Re-review Status

### Original Issues

| # | Original Issue | File | Status |
|---|----------------|------|--------|
| 1 | Memory leak | `src/pool.py:45` | ✅ Fixed |
| 2 | Missing doc | `src/api.py:12` | ❌ Not Fixed |

### New Findings

（变更引入的新问题，格式同首次 review）
```

## 模板 2: Inline Comment（行级评论）

通过 Review API 的 `comments[]` 数组提交。

### 数据结构

```json
{
  "path": "src/pool.py",
  "position": 5,
  "body": "<!--AGENT_INLINE:{\"id\":\"f-1\",\"level\":\"P0\",\"file\":\"src/pool.py\",\"line\":45}-->\n\n❌ **[P0] Memory leak**\n\nConnection not released on exception path. See review body for full details."
}
```

### Body 模板

```markdown
<!--AGENT_INLINE:{"id":"${ID}","level":"${LEVEL}","file":"${FILE}","line":${LINE}}-->

${LEVEL_EMOJI} **[${LEVEL}] ${BRIEF_TITLE}**

${ONE_LINE_SUMMARY}
```

> **设计原则：** Inline comment 保持**极简**，详细内容在 Review Body 中。这样避免 diff 中评论过长，同时确保人类在点击行内评论时能看到足够上下文。

---

## 模板 3: Body-Only Fallback

当 inline comments API 报错时，降级为纯 body 提交。此时将所有 findings 完整展开在 body 中。

与模板 1 的区别：
- 保留 AGENT_META
- 保留 AGENT_FINDING 注释
- 所有 findings 完整展开（不依赖 inline comments 跳转）

## Emoji 规范

| 场景 | Emoji | 说明 |
|------|-------|------|
| P0 Critical | ❌ | 阻塞性问题 |
| P1 Important | ⚠️ | 重要问题 |
| P2 Suggestion | 💡 | 建议 |
| P3 Nitpick | 📝 | 细节 |
| PASS | ✅ | 通过 |
| N/A | ⏸️ | 不适用 |
| Summary | 📊 | 摘要 |
| Review | 🔍 | Review 报告 |
| Re-review | 🔄 | 复核 |
| Fixed | ✅ | 已修复 |
| Disagreed | ❌ | 不同意 |
| Question | ❓ | 疑问/需要澄清 |

## Agent 解析规范

### 正则提取模式

```regex
<!--AGENT_META:({.*?})-->
<!--AGENT_FINDING:({.*?})-->
<!--AGENT_INLINE:({.*?})-->
```

### 解析步骤

1. 提取 `AGENT_META` → 获取 review 级元数据
2. 遍历提取所有 `AGENT_FINDING` → 获取 findings 列表
3. 交叉验证：`AGENT_META.findings.length == AGENT_FINDING.count`
4. 若不一致，以 AGENT_FINDING 为准（更细粒度）

---

## GitHub API 兼容性

- `body` 字段最大长度：**65536 字符**
- 若 review body 超出限制 → 截断 findings 列表，在末尾添加 "_... and N more findings truncated_"
- `comments[]` 数组元素 `body` 最大长度：**65536 字符**（通常不会超限）
- HTML `<details>` 和 `<summary>` 标签在 GitHub 渲染中完全支持
- HTML 注释 `<!-- -->` 在渲染中被完全忽略
